# M4 — External edits & conflict handling

**Goal:** make Brulion a well-behaved *view* on a folder that other tools also
write to (an AI session, a CLI, vinote, a native capture helper). The folder is
the API; the app must reflect what's on disk and must never silently clobber a
change made outside it. See `DECISIONS.md` → "Files are the interface" and the
M4 entries below.

Success: while Brulion is open, a note created/edited/deleted by another tool
shows up in the app on its own; when the on-disk version of the note you're
editing changes under you, you are never silently overwritten — you get a clear
choice to keep your version or take the disk's.

## Progress
- [ ] Phase 1 — Change-detection layer (polling)
- [ ] Phase 2 — Live, non-conflicting refresh
- [ ] Phase 3 — Conflict resolution UX

Phases build bottom-up, mirroring M3. Phase 1 is the pure detection layer: a
poller plus a controller `checkDisk()` that diffs the folder listing and the
active note's `lastModified`, classifying what changed without deciding the UX.
Phase 2 wires the non-conflicting cases — the common path where Brulion is just a
view: external list changes refresh the sidebar, and an external edit to the
active note **with no local unsaved edits** silently reloads the buffer. Phase 3
handles the hard case: an external change collided with local unsaved edits (or a
save was refused) — replace M1's dead-end "stuck conflict" with real recovery.

Constraints that apply throughout:
- **No silent clobber is the moat.** Detection must never lead to overwriting an
  external change the user hasn't seen. When in doubt, surface, don't write.
- **The app is a view, not the owner.** The disk is the source of truth; the
  common (non-conflicting) case should just track it with zero friction.
- **Lean.** Poll, don't depend on experimental watch APIs. Two-way conflict
  choice, no diff/merge UI (see `DECISIONS.md`).

## Phase 1 — Change-detection layer (polling)
- A generic poller (`watch.ts`): `createPoller(tick, intervalMs) → { start, stop }`
  that runs an async `tick` on an interval and **never overlaps runs** (a tick
  still in flight when the timer fires is skipped, not stacked). Testable with
  fake timers.
- A controller `checkDisk()` method that, against the controller's known state,
  re-lists the folder and re-stats the active note's `lastModified`, and
  classifies the result: list added/removed, active-note changed on disk,
  active-note deleted on disk — distinguishing the no-local-edits case from the
  has-local-edits (conflict) case. It only **detects** here; emitting/acting is
  Phases 2–3. `checkDisk()` runs on the controller's existing serialize queue so
  it can't interleave with open/switch/save.
- **Done =** unit tests cover the poller (fires on interval, no overlap, stop is
  clean) and the classification in `checkDisk()` (list diff; active-note mtime
  change with and without local edits; active-note deleted) with the FSA mocked.

## Phase 2 — Live, non-conflicting refresh
- Start the poller when a folder opens; stop/replace it when the folder is
  re-picked. (One poll loop per open folder.)
- An external **list** change (note added or removed by another tool) refreshes
  the sidebar via the existing `onListChanged`, preserving the active note.
- An external **content** change to the active note, **when there are no local
  unsaved edits and no conflict**, silently reloads the editor buffer and updates
  the known `lastModified` — Brulion tracking the disk. The caret/scroll are kept
  reasonable (don't fight the user; this path only runs when they're not mid-edit
  on unsaved changes).
- An active note **deleted** externally **with no local edits** behaves like an
  in-app delete of the active note: switch to another note (or the empty-folder
  state). (The has-local-edits variant is Phase 3.)
- **Done =** with the app open on note A: another writer adding/removing a note
  updates the list; another writer editing A (while A has no unsaved local edits)
  updates the visible content; deleting A externally moves the user off it — all
  without a reload and without losing any *saved* data.

## Phase 3 — Conflict resolution UX
- Define the conflict state precisely: the active note changed (or was deleted)
  on disk **while the buffer has local unsaved edits**, reached either
  proactively (the poller from Phase 1) or reactively (a save refused by the
  existing stale-write guard). Both converge on one state and one UX.
- Replace M1's dead-end (conflict set → editing frozen, no way out) with a clear
  choice surfaced in the UI:
  - **Keep my version** — overwrite the on-disk file with the buffer (re-base on
    the file's current mtime, then write; for an externally-deleted note this
    re-creates it). Clears the conflict.
  - **Use the version on disk** — discard local edits and reload from disk (for a
    deleted note, switch off it). Clears the conflict.
- No diff/merge view (see `DECISIONS.md` → "Conflict UX: two-way choice, no
  diff"). Both options are non-destructive to the *other* side's data until the
  user picks: nothing is written or discarded silently.
- **Done =** with unsaved edits to note A, an external change to A surfaces the
  conflict; "Keep my version" lands the user's text on disk; "Use the version on
  disk" loads the external text and drops local edits; editing is usable again
  after either choice; the same UX handles A being deleted externally mid-edit.

## Out of scope (deliberately)
- Diff/merge UI or three-way merge — a two-way keep/take choice is the lean cut
  (see `DECISIONS.md`).
- `FileSystemObserver` / real OS file-watching — experimental and Chromium-only;
  polling `lastModified` is the portable, lean choice (see `DECISIONS.md`).
- Watching subfolders / nested notes — notes are root-only (M3 decision).
- Conflict handling for non-active notes' buffers — only one note is open at a
  time; background notes have no unsaved local state to conflict.
