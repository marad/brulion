# M19 — Note URLs / history

**Goal:** give every open note its own URL so the browser's Back/Forward becomes
prev/next navigation over visit history, and the URL is a self-bookmark of state.
One mechanism — a hash route mirroring the open note — covers both "go back to
the note I came from" and "bookmark this note". URL-only, moat-neutral: no change
to file behavior.

This is also the foundation **M21** builds on: recency = most-recently-*visited*
reuses the same visit history the browser now records for free.

## P1 — Hash route for the open note

A single hash-route mechanism, three directions, no custom history stack:

- **Open note → URL.** Every active-note change reflects into a `#/path/to/note`
  hash route (no `.md`, segments individually percent-encoded). A genuine
  navigation pushes a history entry; the initial load settles the URL with
  `replaceState` so it leaves exactly one entry for the resolved note.
- **URL → open note.** A `hashchange` (Back/Forward, the mouse back button, an
  edited/pasted URL) switches to the note named by the hash, reusing the existing
  `switchTo`. A hash naming a note that doesn't exist is inert (stay put) — no
  create-on-miss.
- **Load/bookmark.** On load, once the folder is granted, the hash takes
  precedence over the persisted active note; with no hash (or a hash naming a
  missing note) the normal first-run / persisted-active flow stands.

**Done =** opening a note updates the URL hash; Back/Forward navigate visit
history; reloading or opening a bookmarked `#/…` URL lands on that note once the
folder is re-granted; unit tests cover the pure path↔hash codec and an e2e spec
covers real Back/Forward navigation.
