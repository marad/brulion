# M15 — Code-block syntax highlighting

**Goal:** fenced code blocks (FEAT-0016 renders them as a box) get syntax colors for
the popular languages, driven by the fence's info string (```` ```ts ````).
**Editor-only** (CodeMirror language data + a highlight style); the on-disk markdown
is unchanged — purely how the block is painted.

The follow-up agreed in the M23 review rides here as P2: paint the **expanded
frontmatter** region as YAML on the same highlight infrastructure — decoration-only
and opaque (colors tokens, never interprets fields), consistent with M23's moat
stance.

## P1 — Syntax-highlight fenced code blocks by info string

The markdown language already renders a closed fenced block as a box (FEAT-0016) but
leaves its contents plain. This phase wires the markdown parser's nested-language
support so the block's body is parsed by the language its info string names, and adds
a light-theme highlight style so the tokens get colors.

- **`codeLanguages` via `@codemirror/language-data`.** The `markdown(...)` call gains
  `codeLanguages: languages`, so a fenced block whose info string matches a known
  language (or alias) is parsed by that language's grammar. Parsers **load lazily**
  (dynamic import) only when such a block appears — zero cost for code-free notes —
  and the service worker (FEAT-0029) caches each loaded chunk cache-first, so after a
  first online encounter the language highlights offline too. An unknown info string
  (or no info string) leaves the block plain, exactly as today.
- **A focused light-theme `HighlightStyle`.** A `syntaxHighlighting(HighlightStyle)`
  maps the common programming-token tags (keyword, string, number, comment, operator,
  variable/property/type/function names, booleans, regexp, escape, meta) to colors
  tuned for the editor's light background. It deliberately does **not** style the tags
  markdown prose uses (heading/emphasis/strong/link/list), so the existing
  decoration-based prose rendering is untouched.

**Done =** a ```` ```ts ```` (or `js`/`json`/`python`/… ) block shows colored tokens;
an unknown/blank info string stays plain; markdown prose styling (headings, bold,
links) is unchanged; the on-disk bytes are untouched; a unit test asserts the
markdown syntax tree nests the named sub-language over a fenced block, and an e2e
asserts a code block renders multiple colored token spans.

## P2 — Highlight the expanded frontmatter as YAML

The expanded frontmatter region (FEAT-0042) is plain editor lines today. This phase
paints it as YAML, reusing P1's `HighlightStyle`. Because the frontmatter is detected
structurally (not by the markdown grammar), it is highlighted by parsing the region's
text with the YAML parser (`@codemirror/lang-yaml`) and emitting `Decoration.mark`s
carrying the same highlight classes — only while the region is **expanded** (the
collapsed chip shows nothing to color).

- Decoration-only and **opaque**: it colors YAML tokens, never interprets fields, and
  never reads or rewrites the bytes (the M23 moat stance, kept).
- Built into the existing `frontmatterField` expanded branch, so it tracks the
  collapse state and document edits with no new field.

**Done =** an expanded `---…---` block shows YAML token colors (keys, strings,
numbers) consistent with code blocks; the collapsed chip is unaffected; nothing is
hidden or made atomic by the highlight; the bytes are untouched; a unit test asserts
highlight marks land over the YAML tokens in the expanded region and none when
collapsed.
