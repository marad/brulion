# M18 — Light / dark theme

**Goal:** a real light / dark / system theme, surfaced as a preference in the M16
settings modal. Net-new feature, not just a relocated toggle: today there is a single
hard-coded light look (colors scattered as literal hex across `styles.css` and the
editor theme). Editor/UI chrome only — the on-disk markdown is unchanged. Split out of
M16 deliberately so the settings modal could ship on typography + Vim first.

Scope (decided autonomously, recorded in `DECISIONS.md`):

- **A semantic CSS-variable palette** — the scattered hex migrated to a small set of
  tokens (page bg, surfaces, text/muted/faint, borders, accent + accent-strong/fg/text,
  link, code bg, soft-accent surfaces/fills, danger surfaces, shadow). The **light**
  values equal today's colors (near-duplicate shades collapsed to one token), so the
  light look is unchanged; only the dark set is new.
- **Three modes via a `data-theme` root attribute + `prefers-color-scheme`.** `system`
  (default) sets no attribute and lets a media query supply dark on a dark OS;
  `light`/`dark` set `data-theme` to force a palette. JS only sets/clears the attribute.
- **The editor reads the same vars** (`editor.ts` + `markdown-render.ts`), so the
  editing surface themes with the chrome.

Out (deliberately): per-element overrides / custom palettes (three modes, one dark
palette); a theme transition animation; syntax-highlight retuning beyond the shared
tokens.

## P1 — Light / dark / system theme

`Settings` gains `theme: "light" | "dark" | "system"` (default `system`), normalized
and persisted in `.brulion.json`. `applyTheme` toggles the `data-theme` root attribute;
`applySettings` calls it on every settings change. `styles.css` defines the light
palette in `:root` and the dark palette in `:root[data-theme="dark"]` + a
`@media (prefers-color-scheme: dark) :root:not([data-theme])` rule; all chrome colors
reference the tokens, and the editor theme does too. The settings modal gains a
Light / Dark / System radio group. Unit tests cover normalize + `applyTheme` +
the modal radio; e2e covers dark-applies-live + persist, light/system attribute, and
bytes-unchanged.

**Done =** a Light / Dark / System picker in settings re-themes the whole app live
(chrome + editor), `system` follows the OS, the choice persists with the vault, the
light look is unchanged, and no `.md` bytes are touched.

## P2 — Theme-aware code syntax palette

P1 deferred the code-block syntax colors, so fenced blocks still read as light-theme on
the dark code box (a near-black navy for strings, mid-blues for numbers). The seven
distinct `tok-*` colors in `code-highlight.ts` become `--tok-*` palette tokens (light =
the original GitHub-light values, a GitHub-dark-readable override in both dark blocks);
`codeTokenTheme` reads the tokens via `var(--…)`. Shared colors keep one token, so the
light rendering is byte-identical. e2e asserts the worst-case string token is light in
dark mode; the existing scoping/bytes tests still hold.

**Done =** code-block syntax colors flip with the theme — readable on the dark code box
in dark mode, byte-identical in light — with no new switching logic and no `.md` writes.
