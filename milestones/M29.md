# M29 — Reveal code-fence markers when editing inside

**Goal:** make a fenced code block's markers editable. Since M5/FEAT-0016 the opening
```` ```lang ```` and closing ```` ``` ```` lines are **always** hidden, so a typo in
the fence or its info string (e.g. ```` ```mermiad ````, or ```` ```js ```` →
```` ```ts ````) can't be fixed in place. Surfaced during the M28 review. Editor-only,
no file-behavior change.

Scope (decided autonomously, recorded in `DECISIONS.md`):

- **Reveal the fence lines when the selection is inside a closed fenced block**
  (strict overlap, all selection ranges), re-hiding on the way out — the same
  reveal-on-selection pattern as links (FEAT-0026) and Mermaid (FEAT-0056). The change
  lives in `markdown-render.ts` (`blockSyntaxRanges` + `blockRenderingField`, which now
  rebuilds on selection).
- **Keep the code-box styling while revealed** — only the fence *text* un-hides; the
  block still reads as a code box.
- **Applies to all fenced blocks** — Mermaid benefits for free (its revealed source
  now shows its ```` ```mermaid ```` fence).

Out (deliberately): revealing inline-code (`` `x` ``) markers; any change to how the
code body or its highlighting renders.

## P1 — Reveal fence markers on selection

In `blockSyntaxRanges`, skip the two fence-hide ranges for a fenced block any
selection range overlaps; `blockRenderingField` rebuilds on `tr.selection` (was
document-change only). The code-box line styling is pushed unconditionally, so it
persists when revealed. Unit tests cover reveal-inside / hide-outside / box-styling;
e2e covers reveal→edit→re-hide, the Mermaid fence, and bytes-unchanged.

**Done =** a caret inside a fenced block reveals its ```` ```lang ````/```` ``` ````
lines for editing (incl. the Mermaid info string), they re-hide on leaving, the box
styling stays, and the on-disk bytes are unchanged.
