# M21 — Search ranking & recency

**Goal:** fix the quick-switcher ranking (a real reported failure) and add
recency, both centered on `note-search.ts`. Pure logic; no change to file
behavior. Two independent concerns, one per phase.

The ranking is the urgent half: deep paths and clean contiguous runs both rank
worse than they should today. Recency is the comfort half: an empty switcher
should surface where you've just been, and equal-quality matches should break
ties toward the note you touched most recently.

> **Note on the M19 dependency.** The roadmap framed recency as "reuses the M19
> visit history — free". In practice M19 (FEAT-0036) leans entirely on the
> browser's own Back/Forward stack, which JS cannot read back. So recency needs
> its **own** most-recently-visited list — small, persisted alongside the other
> `brulion:` UI state, fed by the same active-note change M19 already mirrors into
> the URL. Cheap, but not literally "free".

## P1 — Ranking fix

Two compounding flaws in `note-search.ts` today, both in `fuzzyScore`:

- **Depth is punished.** The first matched char's "gap" is its absolute distance
  from the string start, so a match deep in a long path
  (`Allegro/Journal/Week/2026-06-22`) pays for every folder character before it.
- **Greedy matching misses contiguous runs.** The matcher grabs the first
  occurrence of each query char left-to-right, so a clean contiguous substring
  (`06-22`) is never recognized *as* contiguous when an earlier scattered
  alignment exists.

Fix, keeping the function pure & total and the `searchNotes` contract intact:

- **A literal contiguous substring wins.** When the query occurs as a
  case-insensitive substring, it scores in a band strictly above any
  non-contiguous (subsequence) match, ranked by *where the run sits* —
  segment-start (after `/`, space, `-`, `_`, or string start) beats mid-token.
- **Best alignment, not greedy.** The subsequence fallback maximizes over all
  alignments (a small DP), so the best-scoring placement is found, not the first.
- **Depth costs nothing.** No leading-distance penalty; only *interior* gaps
  (chars skipped between matched chars) are penalized, so a match's folder depth
  never lowers its score.

**Done =** `searchNotes`/`fuzzyScore` rank a contiguous substring above a
scattered match regardless of path depth; a deep-path note ranks by its name, not
its folder distance; existing switcher/autocomplete callers are unchanged; unit
tests cover the substring-wins, depth-free, and best-alignment cases.

## P2 — Recency: most-recently-visited ordering & tiebreak

A small most-recently-visited (MRU) list of note paths, persisted with the other
`brulion:` UI state and updated on every active-note change, fed into the ranking:

- **Own MRU list.** Maintain an ordered list of visited note paths (most-recent
  first), capped to a sane length, persisted in IndexedDB. Touched whenever the
  active note changes (the same signal M19 mirrors into the URL). Stale entries
  (renamed/deleted notes) are harmless — they are intersected against the live
  note set before use.
- **Empty query → most-recent-first.** With no query typed, the switcher lists
  recently-visited notes first (often no typing needed to hop to a nearby note),
  then the rest in name order.
- **Non-empty query → recency is only a tiebreaker.** Match quality rules; recency
  breaks ties between *equal* scores only — never added into the score, so a
  freshly-touched poor match can never jump a better one.

**Done =** opening the switcher with an empty query lists most-recently-visited
notes first; with a query, recency only reorders equal-scored matches; the MRU
list survives a reload; unit tests cover empty-query ordering and the
equal-score-tiebreak (and that recency never outranks a better match); an e2e
spec covers visiting notes then seeing them recency-ordered in the switcher.
