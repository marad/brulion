# M36 — Keyboard navigation for the sidebar tree

**Goal:** M35 made every tree *action* keyboard-reachable (Shift+F10 context
menu), but *movement* between rows still needs the mouse. This milestone makes
the sidebar a proper keyboard-navigable `tree`: arrow keys move focus between
rows and expand/collapse folders, completing the keyboard story M35 started.

Scope (decided autonomously, recorded in `DECISIONS.md`; reviewed live at the
end):

- **Roving tabindex, the standard ARIA `tree` focus model.** Exactly one row is
  Tab-focusable at a time; arrow keys move focus *within* the tree and carry the
  tab stop with them, so Tab enters the tree at wherever focus last was and one
  more Tab leaves it — not 200 tab stops for a 200-note vault.
- **Movement only, no new actions.** Create/delete/move/rename stay on the
  existing context menu (M35/FEAT-0071). This milestone adds no operation that
  changes a file; it only moves focus and toggles a folder's expansion (which
  M35/FEAT-0043 already persists).
- **Reuse the existing focusables.** A folder row's `.folder-header` and a note
  row's `.note-name` are already `<button>`s; this manages `tabindex` across
  them and wires one keydown handler, rather than a new focus surface.
- **Pure traversal core.** The "given the visible rows and the pressed key,
  what happens" logic lives in a small pure module (`tree-nav.ts`), unit-tested
  without the DOM, mirroring how `link-rewrite.ts`/`note-name.ts` isolate the
  testable core from the FSA/DOM glue.

## P1 — Arrow-key tree navigation

A keydown handler on the tree container drives the standard ARIA `tree`
interaction over the *visible* rows (a collapsed folder's children are in the
DOM but `hidden`, so they are skipped): **Up/Down** move to the previous/next
visible row; **Right** expands a collapsed folder or, if already expanded, moves
to its first child (nothing on a note); **Left** collapses an expanded folder
or, if already collapsed / a note, moves to the parent folder; **Home/End** jump
to the first/last visible row; **Enter/Space** open the focused note or toggle
the focused folder. A roving tabindex keeps exactly one row in the tab order,
updated as focus moves; the active note's row is the initial tab stop (or the
first row when nothing is active). Expand/collapse reuses the same toggle +
persistence (`onToggleFolder`, FEAT-0043) a header click already runs.

**Done =** with focus in the sidebar, the whole tree is navigable by keyboard
alone — Up/Down walk every visible row in order, Right/Left expand/collapse and
descend/ascend folders, Home/End jump to the ends, Enter/Space open a note or
toggle a folder; only one row is ever a Tab stop; and none of this writes a
note file (movement + folder-expansion only).
