# M10 — Welcome / first-run screen

**Goal:** the first thing a new user sees should explain what Brulion is and
invite them in — not a bare blinking editor. Today, before a folder is open, the
app shows an empty CodeMirror buffer and a lone "Open folder" button in an
otherwise empty header. M10 replaces that with a deliberate first-run screen: the
product name, a one-line pitch, the file-fidelity promise stated as a *feature*
("your notes are plain `.md` files on your disk, nothing uploaded"), and a clear
call to open a folder. It also tidies which header controls show before vs after a
folder is open. This is pure UI on the existing foundation — **no change to any
file behavior**; the moat is untouched.

Success: opening the app with no folder yet shows a welcoming, self-explanatory
screen with the open-folder call to action (and the resume-access variant when a
folder was previously granted); picking a folder hides it and reveals the normal
workspace; the header shows only what's relevant in each state; re-picking a
different folder still works once one is open.

## Progress
- [x] Phase 1 — The welcome screen + contextual header (FEAT-0031)

Constraints that apply throughout:
- **Moat untouched.** Nothing here reads, writes, indexes, or caches the user's
  notes. The folder-open / permission / save / poll logic is unchanged; M10 only
  adds a view and moves existing controls between two visibility states.
- **No regression.** Open/restore/resume-access, autosave, the poller, the
  conflict UX, the tree, links, Vim, the sidebar collapse, and the PWA install
  button must all keep working. In particular, re-picking a different folder after
  one is open must stay possible.
- **Lean.** Vanilla TS + CSS, reusing the existing `header button` styling and the
  `hidden` attribute / a CSS class to flip between the pre-folder and post-folder
  states. One small view, no framework, no new dependency.

## Phase 1 — The welcome screen + contextual header
A `#welcome` hero is shown while no folder is open: the **Brulion** name, a short
pitch, a one-line file-fidelity reassurance, and the primary **Open folder** call
to action (plus the **Resume folder access** variant, surfaced exactly when the
existing restore flow would surface it). When a folder opens, the hero is hidden
and the normal workspace (sidebar + editor) takes over. The header gains a
persistent **Brulion** wordmark and shows controls contextually: before a folder
is open only the wordmark (and the Install button when the browser offers it);
after a folder is open the wordmark plus the sidebar toggle (`☰`), Vim toggle, the
Install button, and a re-pick **Open folder** affordance. The open/restore/resume
wiring is the existing one, relocated/extended — not reimplemented.

- **Done =** with no folder open the app shows the `#welcome` hero (name, pitch,
  file-fidelity line, **Open folder** CTA) instead of a bare editor, and the
  **Resume folder access** action appears there in the restore-but-not-granted
  case; picking a folder hides the hero and shows the sidebar + editor; the header
  shows only the wordmark (+ Install when available) before a folder is open and
  adds `☰` / Vim / re-pick **Open folder** after; re-picking a different folder
  while one is open still opens it; a unit test covers the pre-folder ↔ post-folder
  visibility toggle and an e2e covers welcome → open folder → workspace (and the
  hero not reappearing after open).

## Out of scope (deliberately)
- **Workspaces / multiple folders** (`?ws=…`) — still future; M10 keeps the
  single-folder model and only makes re-pick reachable, not multi-folder.
- **Onboarding tour / tooltips / animations** — a static, calm first screen, not a
  guided walkthrough.
- **Empty-folder UX after opening** — the seed `start.md`-on-first-capture behavior
  (FEAT-0004) is unchanged; M10 is about the *pre-folder* state only.
