# M13 — Sidebar comfort: collapsed-by-default tree + resizable width

**Goal:** make the note tree pleasant in a large vault. Two independent tweaks in
the sidebar layer; no change to file behavior.

Phases:

- [x] **Phase 1 — Folders collapsed by default** (`FEAT-0043`)
- [x] **Phase 2 — Drag-to-resize sidebar width** (`FEAT-0044`)

## Phase 1 — Folders collapsed by default

The tree opens collapsed (only the top level shown); the user expands what they
need and that choice persists. Flips the default polarity: today a folder is
expanded unless it is in the persisted *collapsed* set (FEAT-0024); now a folder
is collapsed unless it is in a persisted *expanded* set. The "reveal the active
note's ancestors" rule is preserved — an ancestor of the open note always shows.

**Done =** on first open every folder is collapsed (only top-level rows visible);
expanding a folder persists across reload; collapsing it again persists; the open
note's ancestor folders are always expanded so it is never hidden. No bytes
written to the folder. Covered by unit tests + an e2e test against the real
sidebar.

## Phase 2 — Drag-to-resize sidebar width

Grab the sidebar/editor border and drag to set the sidebar width; the chosen
width persists alongside the other `brulion:` UI state and replaces the fixed
`14rem`. Width is clamped to a sane min/max so the sidebar can be neither hidden
nor allowed to swallow the editor.

**Done =** dragging the border resizes the sidebar live; the width is clamped to
the min/max; the chosen width survives a reload; the resize handle disappears
when the sidebar is collapsed/hidden; no bytes written to the folder. Covered by
a unit test on the clamp logic + an e2e test against the real drag.
