# M25 — Update references when a note is renamed

**Goal:** keep the vault consistent across a rename — links in *other* notes that
point at the renamed/moved note follow it, instead of silently dangling. The M14
follow-up agreed in that milestone's review: rename (FEAT-0034) deliberately
moves only the one file; this milestone closes the loop on inbound links.

This is **not** a moat violation — rewriting a link is still plain markdown the
user owns, and it happens only on an explicit, user-initiated rename, **after the
user confirms which files will change** — but it is a real multi-file write with
genuine edge cases, so it gets its own spec/test/review cycle.

Phases:

- [x] **Phase 1 — Rewrite inbound links on rename** (`FEAT-0040`)
- [x] **Phase 2 — Rebase the moved note's own outbound links** (`FEAT-0041`)

## Phase 1 — Rewrite inbound links on rename

A pure rewrite core (`link-rewrite.ts`) that, given one note's text and the
rename `old → new`, returns the text with every link that pointed at the old path
re-pointed at the new one — handling **markdown relative links** (rebased through
the linking note's folder, so a folder move recomputes the `../` prefix),
**slashed wikilinks** (root-relative path), and **bare wikilinks** (basename,
kept bare when still unambiguous, promoted to a full path only when the move would
make the bare name resolve to a different note). It reuses the existing resolvers
(`resolveNotePath` / `resolveWikilink`) as the single source of truth for "what
does this link point at", and `shortestLinkText` for the bare-vs-full wikilink
choice. The controller's `renameActive` drives it across the vault: after the
file moves, it scans every other note, builds the set of edits, asks the UI to
**confirm which files will change**, then writes each through the existing
per-note conflict guard (`saveNote` with the file's known mtime) so a note edited
out from under us is skipped, never clobbered.

**Done =** renaming a note **silently** rewrites markdown and wikilink references
to it in other notes so they still resolve to the new path (bare wikilinks stay
bare when safe); a write that would clobber an externally-changed file is skipped;
the pass is best-effort (its failure never fails the rename) and the renamed
file's own bytes are untouched by it. Pure rewrite covered by unit tests; the
multi-file rename covered by an e2e against the real FSA (OPFS). *(The
confirmation prompt from the first cut was removed in the M25 review — see
`DECISIONS.md`.)*

## Phase 2 — Rebase the moved note's own outbound links

The mirror of Phase 1, surfaced in the M25 review: when a rename also moves the
note to a **different folder**, its *own* outbound relative markdown links (which
were resolved relative to the old folder) would point at the wrong place. A pure
`rebaseOutboundLinks(text, old, new)` recomputes each in-tree relative markdown
destination from the new folder (reusing `relativeLink`/`resolveNotePath`,
round-trip-correct); wikilinks, external, and non-note links are left alone, a
self-link follows the note, and a same-folder rename changes nothing. The
controller runs it after the move, before re-pointing the editor, through the same
stale-write guard.

**Done =** moving a note to another folder rebases its own relative markdown links
so they still resolve to the same targets (a same-folder rename changes nothing);
wikilinks and external links are untouched; a self-link follows the note. Pure
rebase covered by unit tests; the folder-move case covered by an e2e against the
real FSA (OPFS).
