# M9 â€” PWA

**Goal:** make Brulion installable as an app and usable offline. Open the URL
once on a machine, install it to the dock/start menu, and from then on it
launches in its own window and loads even with no network â€” because the app
shell (HTML/JS/CSS) is cached. The user's notes are untouched by any of this:
they still live as plain `.md` files on disk via the File System Access API, and
the service worker only ever caches *our* static assets, never the user's data.
Offline simply means the editor opens; reaching the user's folder still needs the
folder to be present and access re-granted exactly as today.

Success: the app meets the browser's installability bar (manifest + icons +
offline-capable service worker), can be installed from a custom in-app affordance
(not only the browser's address-bar default), launches standalone, and the shell
loads after going offline.

Constraints that apply throughout:
- **The moat is untouched.** The service worker caches only the app's own static
  build output. It never sees, caches, or mediates the user's `.md` files â€” those
  are read/written through the File System Access API, not `fetch`. Nothing here
  adds an index, a sidecar, or app-private copies of the user's data.
- **Lean.** No PWA build plugin / Workbox. A hand-rolled service worker of a few
  dozen lines, a static manifest, and a small registration call. Reuse the
  existing `idb-keyval`-free, vanilla-TS shape; add the smallest thing that holds.
- **Don't regress M1â€“M8.** Folder open/permission re-grant, autosave + conflict
  UX, the external-change poller, the tree, links, and Vim must all keep working
  identically. The SW must not interfere with the FSA paths or stale-serve the app
  across deploys.
- **Base path.** GitHub Pages serves the site under `/brulion/`. The manifest,
  icons, service worker URL, and its scope all live under that base
  (`import.meta.env.BASE_URL`), not the domain root.

## Phase 1 â€” Web App Manifest + icons
A `manifest.webmanifest` makes the app describable to the OS: `name` /
`short_name`, `description`, `display: standalone`, `theme_color` /
`background_color`, `start_url` and `scope` under `/brulion/`, and an `icons`
array. Icons are generated from a single simple source mark (a notepad glyph) at
the install sizes (192 and 512), including a `maskable` variant so the installed
icon isn't letter-boxed. The manifest is linked from `index.html` and the
matching `theme-color` meta and apple-touch-icon are added.

- **Done =** `manifest.webmanifest` exists with valid `name`/`short_name`,
  `display: standalone`, theme/background colors, `start_url`+`scope` resolved
  under the Pages base, and an `icons` array referencing real icon files at 192
  and 512 (with a maskable variant); `index.html` links the manifest and sets
  `theme-color`; the icon files ship in the build output; a unit test parses the
  manifest and asserts the required fields and that every referenced icon file
  exists. (Installability isn't fully met until Phase 2 adds the SW â€” that's
  expected; this phase delivers the metadata half.)

## Phase 2 â€” Service worker: offline app shell
A hand-rolled service worker caches the app shell so the editor loads with no
network. Strategy split by request type: **navigations (the HTML document) are
network-first with a cache fallback** â€” a fresh deploy is picked up while online,
but the last-cached shell still opens offline; **hashed static assets are
cache-first** â€” their content-hashed filenames make them immutable, so serving
from cache is always correct and a new deploy fetches new URLs. The cache is
versioned and old versions are pruned on `activate`, with `skipWaiting` +
`clients.claim` so an update takes effect promptly. The SW is registered from
`main.ts` at `BASE_URL + sw.js` with scope `BASE_URL`. Requests the SW shouldn't
touch (cross-origin, non-GET) pass straight through. The File System Access API
never goes through `fetch`, so the user's notes are categorically outside the
SW's reach.

- **Done =** a `sw.js` ships at the site root-under-base and is registered on load
  (guarded for browsers without `serviceWorker`); after one online visit, the app
  shell (document + JS/CSS) loads with the network offline; navigations are
  network-first (a redeploy is reflected on next online load) and hashed assets are
  cache-first; the activate handler deletes stale cache versions; non-GET and
  cross-origin requests bypass the cache; an e2e test loads the app, goes offline
  (Playwright `context.setOffline`/route block), reloads, and asserts the editor
  still mounts; the user's FSA read/write paths are unaffected (existing e2e stay
  green).

## Phase 3 â€” Install prompt UX: beforeinstallprompt
With the manifest (Phase 1) and offline SW (Phase 2) in place the app is
installable, so Chromium fires `beforeinstallprompt`. We capture it
(`preventDefault`, stash the event), reveal a custom **Install** affordance in the
header (hidden by default, like the other header controls), and on click call the
saved event's `prompt()`. After the choice (or `appinstalled`) the affordance
hides and the stashed event is cleared. When the app is already running installed
(`display-mode: standalone`) the button never shows. The capture/show/clear
decision logic is a small pure unit, with the DOM/event wiring as a thin adapter
(mirroring the FEAT-0020/0021 toggle pattern).

- **Done =** `beforeinstallprompt` is captured and its default mini-infobar
  suppressed; a header **Install** button, hidden until the event arrives, appears
  only when installable and not already in standalone; clicking it calls the
  deferred event's `prompt()` and then hides; `appinstalled` (or a dismissed
  choice) clears the stashed event and hides the button; unit tests cover the
  pure show/hide/clear state logic across the events, and an e2e asserts the button
  is absent before the event and shown after a synthesized `beforeinstallprompt`.

## Out of scope (deliberately)
- **Background sync / push notifications** â€” no backend, nothing to sync or push;
  the SW exists only for offline shell loading.
- **Caching or syncing the user's notes** â€” the notes are FSA files on disk, not
  app assets; putting them in a SW cache would create an app-private second copy
  that drifts from disk, exactly the index/sidecar the moat forbids.
- **Periodic background update / precise update toasts** â€” network-first
  navigation already self-heals on the next online load; a "new version available,
  reload?" toast is separable polish, not part of "installable + offline".
- **iOS-specific install instructions UI** â€” iOS Safari has no
  `beforeinstallprompt`; the manifest + apple-touch-icon make manual
  "Add to Home Screen" work, and a guided iOS flow is PMF-gated polish.
