# M38 — Cross-device permalinks (name-keyed workspaces)

A narrow extension of **M19** (note-URLs `#/path/to/note`) and **M33**
(workspaces `?ws=`): make the full permalink resolve to the *same* note on any of
the user's **own** devices. Today `?ws=` carries a random per-machine opaque id
(`vaults.ts`), so the same synced folder gets a different `?ws` on every device.
Key the workspace by a **stable name** (from `.brulion.json`, which already
travels with the folder) and the URL becomes portable. Moat-neutral — no `.md`
bytes touched; the note is still addressed by its plain path.

See `ROADMAP.md` → M38 and `DECISIONS.md` → "Cross-device permalinks" for scope
and the honest limits (cross-own-devices not foreign-machine; path-permalink not
rename-stable id; a personal bookmark not a sharing link).

## Design spine (settled here; details at spec time)

- **Effective workspace name** = the `workspace` field in `.brulion.json` when
  set, else the folder name (`handle.name`). Defaulting to the folder name means
  portability works out of the box for the common case (one folder, same name
  everywhere); the override is for when local folder names differ across devices.
- **The name is cached in the `Vault` record** (`workspace?: string`), refreshed
  on every attach (when we hold permission and read settings). Startup resolution
  matches `?ws` against the cached effective name **without touching disk** — so
  it works before a handle's permission is re-granted.
- **`?ws` resolution is name-first, id-fallback.** `?ws=notes` → the granted
  vault whose effective name is `notes`; an unmatched value is retried as a legacy
  opaque id (`getVault`) so existing `?ws=<id>` links keep working. Collision
  (>1 match) → most-recent-first wins (the set is ordered that way).
- **The stamped `?ws` is the effective name** (portable), not the opaque id.
- Session storage stays keyed by `vault.id` (unchanged) — only the URL param
  changes meaning.

### P1 — Name-keyed vault resolution

The core. Settings gains a `workspace` field; the `Vault` record caches the
effective name; `?ws` resolves name-first with id fallback; the stamp uses the
name. No new UI (the name defaults to the folder name, so P1 alone already makes
same-folder-name setups portable).

**Done =** on a device that has already granted folder *F* (workspace name `N`),
opening `?ws=N#/path/to/note` attaches to *F* and opens that note; a legacy
`?ws=<oldOpaqueId>` still resolves; an unknown `?ws` falls back to the
most-recent vault (or welcome), preserving the `#/…` hash so the note opens once
the right folder is granted; the URL a window stamps carries the effective name.
Unit tests cover: settings `workspace` normalization; the resolver (name match,
legacy-id fallback, collision → most-recent, miss); the cache write on attach.

### P2 — Workspace name in the settings modal

Surface the `workspace` field in the M16 settings modal so the user can set an
explicit name (needed when a folder is named differently across devices). Empty
shows the folder-name default as a placeholder. Saving writes `.brulion.json`,
refreshes the cached name, and re-stamps the current window's `?ws`.

**Done =** the settings modal shows a "Workspace name" field; typing a name and
saving persists it to `.brulion.json`, updates the cached vault name, and updates
the current URL's `?ws` to the new name; clearing it falls back to the folder
name. A vitest DOM test drives the field → save → persisted-and-restamped path.
