# M11 — Vim caret respects hidden markup

**Goal:** fix a Vim-mode defect — the Vim cursor can land on characters that are
visually hidden (a heading's line-start `#`, a blockquote `>`, a list marker), so
navigation stops on glyphs the user can't see. The default (non-Vim) caret
already steps over these because CodeMirror's own motions honor the editor's
atomic ranges; Vim computes its motions by raw character offset and ignores them.
This milestone makes the Vim caret obey the same rule. Editor-layer fix only —
**no change to any file behavior**; the moat is untouched.

Success: with Vim on, motions never leave the caret inside a hidden markup run or
before a line's leading hidden marker — `0`/`^`/`I` land on the first visible
character, so inserting at the line start goes *after* the `# `/`> `/`* ` marker,
not before it — for headings, blockquotes, and list markers; the default caret and
every existing editor command keep working unchanged.

## Phase 1 — Snap the caret out of hidden runs
A transaction filter watches selection-setting transactions (which is how the Vim
layer moves the caret) and, when an endpoint lands strictly inside a hidden markup
run, snaps it to the nearest edge — forward to the run's end when the motion went
right, back to its start when it went left. The hidden runs are computed from the
same pure functions the renderer uses, scoped to the endpoint's line, so the rule
matches exactly what is on screen. Pointer-driven selections are left alone, so a
click can still place the caret inside a link to reveal its markup. For the
default caret the filter is a no-op (CodeMirror already kept it out of atomic
ranges), so it acts as a shared invariant rather than a Vim-only patch.

- **Done =** with Vim on, `0`/`^`/`I` (and other motions) never leave the caret
  inside or before a hidden `# `, `> `, or `* `/`- ` run — they land on the first
  visible character, so insert-at-line-start goes after the marker; the default
  caret, link reveal, slash/format/Enter commands, and visual selection are
  unchanged; unit tests cover the snap logic and an e2e covers line-start motions +
  insert across a heading/blockquote/list line.

## Out of scope (deliberately)
- **Revealing hidden markup on the Vim cursor line** — no Obsidian-style
  reveal-on-cursor; the markup stays hidden, the caret just doesn't sit inside it.
- **Operator/edit semantics over hidden runs** (`d`/`c`/`x` precisely trimming
  markup) — this is about caret *resting* position, not redefining operators.
- **Inline emphasis/code marks** beyond confirming they aren't regressed — the
  reported pain is line-start block markers; the fix covers any hidden run it sees
  but the ACs target the reported constructs.
