# M17 — Mobile UX

**Goal:** make Brulion usable on a touch device with a narrow viewport, where there
is no Ctrl, no keyboard chords, and Vim is impractical. Two gaps to close: the
side-by-side layout doesn't fit a narrow screen, and formatting is reachable only via
right-click / Ctrl+B-I-E. Everything else already has a visible, tappable affordance
(find/create = the sidebar search button, settings = the gear, sidebar = the ☰
toggle; Vim is off by default and lives in Settings), and saving is autosave +
flush-on-hide — so "graceful absence of the keyboard-only features" is already met
once those buttons are reachable.

**Scope note (flagged for the milestone review):** Brulion's premise is the File
System Access API, which today runs on desktop/Chromebook/convertible Chromium, not
on phone browsers. M17 makes the app **touch-and-narrow-viewport usable wherever FSA
runs** (touchscreen laptops, tablets/Chromebooks in tablet mode, and phones if/when
FSA lands there); it does **not** add a non-FSA storage fallback (that would dilute
the file-fidelity moat) — the no-FSA case still degrades to the welcome screen.

## P1 — Responsive layout & touch sidebar drawer

At a narrow breakpoint the sidebar + editor side-by-side doesn't fit. Below it, the
sidebar becomes a **slide-over drawer** over a full-width editor, dismissable by a
backdrop tap; the drag-resize handle is hidden (irrelevant on touch); the header
controls stay tappable and don't overflow. Desktop layout is unchanged.

The drawer reuses the existing sidebar-collapsed state (FEAT-0020): "not collapsed" =
drawer open (overlaid), "collapsed" = drawer closed — so the ☰ toggle drives it
unchanged. A backdrop tap collapses it, and selecting a note collapses it (expected
drawer behavior) — both only in the narrow layout.

**Done =** at a narrow viewport the sidebar overlays the editor as a drawer with a
dismissing backdrop and the editor spans full width; ☰ opens/closes it; tapping a
note or the backdrop closes it; the resize handle is hidden; the header controls fit
without overflow; at desktop widths the layout and behavior are exactly as before; an
e2e drives a narrow viewport and asserts the drawer/overlay/backdrop/close-on-select
behavior, and a desktop-width e2e asserts the side-by-side layout is unchanged.

## P2 — Touch selection formatting toolbar

Formatting today is right-click (FEAT-0008 context menu) or Ctrl+B/I/E — neither
available on touch. This phase adds a small **floating toolbar that appears over a
non-empty selection** offering the core format actions (bold, italic, code, H1–H3,
clear), reusing the exact FEAT-0007 transforms the menu and shortcuts use (one source
of truth, same clean-markdown result). It works for a touch selection and a mouse
selection alike, and complements (does not replace) the right-click menu.

**Done =** selecting text shows a floating toolbar near the selection; tapping an
action applies the same transform as the shortcut/menu and the selection/markdown is
correct; the toolbar hides when the selection is empty or the editor blurs; it never
fights the right-click menu or the slash menu; an e2e selects text on a touch-like
interaction and asserts the toolbar applies bold/heading and that the on-disk markdown
matches.
