# M8 — Links & subfolders

**Goal:** stop treating the folder as a flat list of names. Both halves are the
same domain — **paths, not just names**. A note stops being a bare filename in
the root and becomes a folder-relative path; the sidebar grows a tree; and notes
can link to each other by path. The moat is unchanged: the markdown files in the
folder tree *are* the API — no index, no sidecar, no app-private state; the
on-disk structure (folders + files) is the single source of truth, and links are
real CommonMark, portable to any other tool.

Success: the user can keep notes in subfolders, see them as a tree in the
sidebar, create a note inside a subfolder, and link from one note to another
(rendered as a clickable link, the markup hidden like every other construct);
following a link switches notes; a broken link is visible as broken.

> Phases 4–5 were added from the **M8 review**: Phase 3 shipped only the bare
> mechanism (`[text](url)`, Ctrl/Cmd+click to follow), which is too thin —
> bare URLs weren't clickable, following was undiscoverable, links were
> near-impossible to edit (hidden URL), and wikilinks (a de-facto standard in the
> plain-markdown note niche) were missing. These phases close that.

Constraints that apply throughout:
- **File fidelity is the moat.** No index file, no metadata sidecar — the folder
  tree is the source of truth, exactly as M3 settled for the flat case. Links are
  standard markdown (`[text](path.md)`), real bytes a CommonMark tool resolves
  the same way; we never invent a non-portable syntax (the underline / wikilink
  trap).
- **Don't regress M3–M7.** Open/switch/create/delete, the autosave guard, the
  external-change poller, and the conflict UX must all keep working — now keyed
  by path instead of bare filename.
- **Lean.** Reuse what's there: `listNotes` recursion over the same `values()`
  iterator, the existing `idb-keyval` persistence, the markdown rendering layers,
  and the controller's serialize queue. Add the smallest thing that holds.

## Phase 1 — Path-addressed storage layer
The note's identity changes from a root filename to a **folder-relative POSIX
path** (`projects/diablo.md`). `note.ts` must traverse subdirectories:
read/save/stat/create/delete resolve a path through intermediate
`getDirectoryHandle` calls (creating them on create/save), and `listNotes`
recurses the whole tree, returning sorted relative paths. `note-name.ts` becomes
path-aware: it normalizes a possibly-pathed name (`projects/Diablo builds` →
`projects/Diablo builds.md`), validating every segment and refusing empty
segments, unsafe characters, and any attempt to escape the root (`..`). The
controller keeps driving the active note as an opaque string — now a path — so
open/switch/create/delete/conflict/poll all work unchanged on paths.

- **Done =** `note.ts` addresses notes by folder-relative path: nested
  read/write/stat/create/delete traverse (and, on create/save, materialize)
  intermediate folders; `listNotes` returns every `.md` in the tree as a sorted
  relative path; `normalizeNoteName` accepts `folder/name`, validates each
  segment, and rejects `..`/unsafe/empty segments; the controller and poller run
  on paths with no behavioral regression; unit tests cover nested
  read/write/list/create/delete and path normalization (including rejection
  cases).

## Phase 2 — Folder tree in the sidebar + nested create/delete
With Phase 1 returning relative paths, the sidebar renders them as a **nested
tree**: each folder is a header, its notes indented beneath it, folders
collapsible with the collapsed set persisted (reusing the `idb-keyval` layer).
The new-note input accepts a path (`projects/diablo`) and creates the note in
that subfolder. The active note's row is highlighted wherever it sits.

- **Done =** the sidebar shows the recursive listing as a tree (folder headers,
  notes nested beneath, root notes at top level); folders collapse/expand and the
  state persists across reloads; typing `folder/name` in the new-note field
  creates the note inside that (possibly new) subfolder and opens it; deleting a
  nested note works and an emptied folder disappears from the tree on the next
  render; the active note is marked wherever it lives; unit tests cover the
  tree-building from a flat path list and e2e covers create-into-subfolder +
  switch + delete.

## Phase 3 — Links between notes
Notes link to each other with **standard markdown inline links to local `.md`
files**: `[text](relative/path.md)`. The renderer hides the link markup (like
every other construct) and styles `text` as a link; **Ctrl/Cmd+click follows it**,
switching the editor to that note (resolved relative to the current note's
folder). A link whose target doesn't exist renders as broken (distinct style) and
following it offers to create the note at that path. `http(s)` links open
externally and are not treated as in-app navigation. Display-only: the bytes
stay real, portable markdown.

- **Done =** a `[text](path.md)` link renders with markup hidden and `text`
  styled as a link; Ctrl/Cmd+click on it switches to the resolved note (relative
  to the linking note's folder); a missing target renders distinctly and
  following it offers to create that note; `http(s)://` links open in a new tab
  and never hijack note navigation; the file keeps the literal markdown link
  bytes; unit tests cover link parsing + relative resolution + missing-target,
  e2e covers click-to-navigate between two notes.

## Phase 4 — Link interaction & editing
Make the links from Phase 3 actually usable. **Plain click follows** a link
(external → new tab, internal → switch/create); **Ctrl/Cmd+click places the
caret** instead (the edit escape hatch), and the cursor reads `pointer` over a
link but a text caret while the modifier is held. **Bare web URLs autolink** —
a typed `http(s)://…` or `www.…` renders clickable (emails stay plain text).
A link's markup **reveals for inline editing when the caret is within it** (a
scoped exception to "always hide" — the URL is content, not noise), and a
**hover tooltip shows the link's target**. External links open reliably in a new
tab (no fragile `window.open` features string).

- **Done =** clicking a link follows it (external opens a new tab, internal
  switches/creates); Ctrl/Cmd+click places the caret in the link without
  following; the cursor is `pointer` over a link and a text caret while
  Ctrl/Cmd is held; bare `http(s)://`/`www.` URLs render as clickable external
  links while emails stay plain text; putting the caret in a link reveals its raw
  `[text](url)` for editing and re-renders on leaving; hovering a link shows its
  target; unit tests cover autolink rendering + the caret-reveal range logic, e2e
  covers plain-click-follow, mod-click-caret-reveal, and an autolinked bare URL.

## Phase 5 — Wikilinks
Add `[[note]]` (and `[[note|alias]]`) on top of Phase 4's interaction model.
A bare `[[name]]` resolves by **basename across the whole tree,
case-insensitively** (`[[DiaBlo]]` finds `projects/diablo.md`); a `[[sub/name]]`
with a slash resolves as a path; ambiguous names pick the first by sorted path.
A missing target renders broken and following it creates the note (a bare name at
the root, a pathed one at that path). Wikilinks aren't CommonMark, so they're
detected by a scan alongside the tree, and reuse Phase 4's follow / reveal /
broken-styling / create paths. The files keep literal `[[…]]` bytes — the
user's chosen trade for ergonomics over strict CommonMark portability (see
DECISIONS, reversing the Phase 3 "no wikilinks" call).

- **Done =** `[[name]]` and `[[name|alias]]` render with the brackets (and the
  `name|` of an alias) hidden and the label styled as a link; a bare name
  resolves case-insensitively by basename anywhere in the tree, a slashed one as a
  path, ties broken deterministically; a missing target renders broken and
  following it creates the note (bare → root, pathed → that path); following an
  existing one switches to it; unit tests cover wikilink parsing + case-insensitive
  basename resolution + alias + missing-target, e2e covers click-to-navigate via a
  wikilink and create-on-missing.

## Out of scope (deliberately)
- **Renaming / moving notes between folders** — create + delete already exist;
  a move/rename UI is separable polish, not part of "paths not names".
- **Drag-and-drop reorder or move in the tree** — the tree is display + navigate,
  not a file manager.
- **Backlinks / a link graph / link autocomplete** — navigation only; the graph
  is a separate, PMF-gated concern (links were nearly cut from the MVP entirely).
- **Empty-folder creation as a first-class action** — folders come into being by
  creating a note inside them, and vanish when emptied; no standalone "new folder".
