# M14 — Open-note identity in the header + rename

**Goal:** always show *which* note is open, and let it be renamed in place. The
two pair up — the header name is the rename affordance. Moat-relevant: a rename
is a real file move, so it must be atomic-ish and never lose content.

Phases:

- [ ] **Phase 1 — Move on disk + rename in the controller** (`FEAT-0034`)
- [ ] **Phase 2 — Header identity + inline rename UI** (`FEAT-0035`)

## Phase 1 — Move on disk + rename in the controller

The file-fidelity core. A storage-layer `moveNote(dir, from, to)` that moves a
note's file using the native `FileSystemFileHandle.move()` (atomic, preserves
the bytes exactly — no read+write+delete), and a `NoteController.renameActive`
that drives it: flush the open note first, move it, then make the new path the
active note and refresh the list.

**Done =** `moveNote` moves/renames a note (creating destination folders),
refuses to clobber an existing target, and reports a missing source; the active
note can be renamed through the controller — content preserved, active state and
list updated, dirty edits flushed first, a standing conflict or an invalid name
refused. All covered by unit tests.

## Phase 2 — Header identity + inline rename UI

The open note's name + path shown in the header, and clicking the name turns it
into an inline editor that renames via the controller. Validation feedback on a
bad name; reuses `normalizeNoteName`. Wired so the header identity tracks the
active note through `onListChanged`.

**Done =** the header shows the open note's path/name; clicking it lets the user
rename in place (Enter commits, Esc cancels); a valid rename moves the file and
the header + sidebar + editor all reflect the new path; an invalid name surfaces
an inline error without renaming. Covered by an e2e test against the real FSA
(OPFS) move path.
