# M20 — Link autocomplete

**Goal:** suggest existing note links while typing, reusing the existing search
ranking. Follow-up to M8 (links/wikilinks) and M12 (the quick switcher's scoring).
Editor-only: no change to file behavior.

The point is one source of truth twice over: the suggestions are ranked by the
**same `note-search.ts` scoring** the Ctrl+K switcher uses, and the candidate
notes come from the **same `linkContext` facet** that already drives valid-vs-broken
link rendering. Typing `[[` opens the list; accepting inserts `[[path]]` and closes
the brackets.

## P1 — Wikilink autocomplete on `[[`

Built on `@codemirror/autocomplete` (the slash menu's machinery), registered as a
second markdown completion source beside the slash source:

- **Trigger on `[[`.** When the caret follows `[[` plus an optional partial target
  (no `]`/newline between), a completion list of existing notes opens.
- **Ranked by the switcher's scoring.** The list reuses `note-search.ts`
  (`searchNotes`) against the partial target, over each note's display path; the
  candidate set is the editor's `linkContext.notePaths` facet — one source of truth
  for "which notes exist", consistent with link rendering and the switcher order.
- **Accept inserts `[[path]]`.** Enter/Tab/click replaces the partial target with
  the note's full display path (`.md` stripped), closes `]]` (reusing an existing
  `]]` if present), and leaves the caret after it.
- **Existing notes only.** No create-on-miss; typing a new name is allowed and
  becomes a dangling wikilink (FEAT-0027 already handles missing targets).

**Done =** typing `[[` in the editor offers a ranked list of existing notes;
accepting one inserts a well-formed `[[path]]` resolving to that note; the order
matches the quick switcher; unit tests cover the source (trigger, ranking, insert
+ bracket-closing) and an e2e spec covers the real `[[`→accept flow in Chromium.
