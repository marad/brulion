# M30 — Command palette + customizable action bar

**Goal:** a keyboard-first way to run *actions* (not just open notes), and a
configurable top bar. A command palette — the same overlay pattern as the M12
Ctrl+K note switcher — lists the available **actions** (go to a note, change
folder, toggle Vim, …) for fuzzy-search + run. Actions are a first-class concept:
each can be **pinned to the top bar** (configured in the M16 settings) and given
an **icon** (from the M27 Lucide set) shown on both its toolbar button and the
palette row. Pure UI/UX over existing capability; **no file-behavior change**.

Scope (decided autonomously, recorded in `DECISIONS.md`):

- **Action model** — a small `Action { id, label, icon?, run() }` plus a registry
  built in `main.ts` from the app's existing capabilities. Folder-switch and the
  Vim toggle (the M30 bullet-3 "migrate onto the action model") become registered
  actions, so they're reachable from the palette and pinnable to the bar.
- **Command palette** — a new `command-palette.ts` overlay that reuses the
  switcher's *pattern* (backdrop, keyboard nav, focus restore) and the existing
  `fuzzyScore`, but not the note-specific switcher module (no create-on-miss, no
  recency). Opened with `Ctrl/Cmd+Shift+P`; fuzzy-filters action labels, Enter
  runs the highlighted action, Esc closes.
- **Customizable action bar** — a header group rendered from an ordered list of
  pinned action ids, each as an icon+label button; an M16 settings surface picks
  which actions are pinned and in what order, persisted in `.brulion.json`. The
  settings gear stays a fixed anchor (never pinnable/removable) so the user can't
  lock themselves out of the surface that configures the bar.

Out (deliberately): per-action keybinding customization; a plugin/extension action
API; pinning the always-present controls (sidebar toggle / settings gear) — the bar
is additive, not a replacement for the fixed chrome.

## P1 — Action registry + command palette

A new `src/command-palette.ts` overlay (its own hidden DOM in `index.html`),
mounted like the quick switcher: it owns no state, takes a `getActions()` dep and
runs the selected action's `run()`. Fuzzy-matching reuses `fuzzyScore` from
`note-search.ts` against each action's label. Keyboard nav (↑/↓/Enter/Esc), focus
restore, and backdrop-click-to-close mirror the switcher. The action registry
lives in `main.ts`, wiring existing capabilities (go to note → `switcher.open`,
switch folder, toggle Vim, toggle note list, open settings); folder-switch and Vim
become first-class actions here. `Ctrl/Cmd+Shift+P` opens it (capture-phase, gated
on workspace-shown + conflict-hidden + not stacking another modal).

**Done =** `Ctrl/Cmd+Shift+P` opens a palette listing the registered actions;
typing fuzzy-filters them; ↑/↓ + Enter runs the highlighted action (and closes);
Esc/backdrop close it; the Vim toggle and folder-switch are runnable from the
palette; no file behavior changes.

## P2 — Customizable action bar + settings pinning

A `Settings.actionBar: string[]` (ordered pinned action ids) added to the M16
model + `normalizeSettings` (drops unknown ids, dedups). A header action-bar group
renders those ids as icon+label buttons (icons from the M27 Lucide set, via the
registry), placed beside the fixed controls. The settings modal gains a section to
pick + order the pinned actions; changes persist to `.brulion.json` and the bar
re-renders live.

**Done =** the settings modal lets the user choose which actions are pinned to the
top bar and in what order; the header renders those as icon+label buttons that run
the action on click; the choice persists in `.brulion.json` and travels with the
vault; unknown ids in a hand-edited file are ignored; no file behavior changes.
