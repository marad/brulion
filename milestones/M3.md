# M3 — Multiple notes

**Goal:** list, create, switch, and delete notes. A plain UI on top of the
finished M1 (pipeline) + M2 (editor) foundation. The user names notes themselves.

Success: the folder can hold many `.md` notes; the user can see them all, pick
which one to edit, add new ones with names of their choosing, and remove ones
they no longer want — and every note is still a plain markdown file they own.

## Progress
- [x] Phase 1 — Multi-note storage layer
- [x] Phase 2 — Note list & switching
- [ ] Phase 3 — Create & delete notes

Phases build bottom-up: the storage layer (Phase 1) generalizes the single
hardcoded `start.md` into named-note read/write/list/create/delete, so the UI
phases have a clean data contract to sit on. Phase 2 puts a list on screen and
makes switching safe (the open note's pending edits flush before another loads).
Phase 3 adds the two mutating affordances (new / delete), which are thin wiring
over the Phase 1 operations.

Constraints that apply throughout:
- **File fidelity is the moat.** A note is exactly a `.md` file in the folder; the
  filename *is* the note name. No index file, no metadata sidecar, no database —
  the folder listing is the source of truth.
- **The user names notes.** No auto-naming from content (see `DECISIONS.md`). The
  empty-folder seed note stays `start` (M1 behavior), created on first capture.
- **Notes live in the folder root.** No subfolders/nesting in M3 (lean); revisit
  if it bites.

## Phase 1 — Multi-note storage layer
- Generalize `note.ts` from the hardcoded `start.md` to read/write a note **by
  name**: `readNote(dir, name)`, `saveNote(dir, name, content, knownMtime)` (the
  no-silent-clobber guard is preserved per note).
- Add `listNotes(dir)` — enumerate the folder's `*.md` files, sorted
  case-insensitively; `createNote(dir, name)` — create an empty file, refusing if
  one already exists; `deleteNote(dir, name)` — remove the file.
- Name handling is pure and unit-tested: normalize a user-typed name to a
  filename (trim, ensure a single `.md` extension), reject empty names and names
  containing path separators or other unsafe characters.
- **Done =** unit tests cover list (sorted, only `.md`), create (and the
  already-exists refusal), delete, read/write by name with the conflict guard,
  and name normalization/validation.

## Phase 2 — Note list & switching
- A note list panel beside the editor renders the folder's notes (display name =
  filename without the `.md` extension); the active note is highlighted.
- Clicking a note switches the editor to it. The currently open note's pending
  edits are **flushed first** (no lost keystrokes), then the selected note loads
  and the save state (mtime/dirty/conflict) resets to that note.
- The active note is persisted (IndexedDB) and restored on reload: reopening the
  folder returns to the last-edited note if it still exists, else the seed
  `start` / first note.
- An empty folder still behaves like M1: the editor opens a `start` buffer and
  `start.md` is created on first capture, then appears in the list.
- **Done =** opening a folder with several notes lists them; clicking switches
  content; edits to note A are saved before switching to B; reload reopens the
  last-active note.

## Phase 3 — Create & delete notes
- A "new note" affordance prompts for a name, creates the file via Phase 1
  (rejecting duplicates / invalid names with a clear message), adds it to the
  list, and switches to it.
- A per-note delete affordance asks for confirmation, removes the file, drops it
  from the list, and switches to another note (or the empty-folder state if it
  was the last one).
- **Done =** the user can create a named note and immediately edit it, and delete
  a note with confirmation; both changes are reflected on disk and in the list.

## Out of scope (deliberately)
- Renaming notes — separable; revisit after the create/delete core lands.
- Subfolders / nested notes, tags, search, sort options beyond alphabetical.
- Links between notes — backlog (see `DECISIONS.md`).
- External-edit detection / live folder watching — M4. The list reflects what was
  on disk at open / after the app's own mutations; it does not poll.
