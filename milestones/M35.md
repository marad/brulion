# M35 — Folder & note management

**Goal:** the sidebar tree can list/select/toggle folders and
create/delete/rename **notes**, but there is no way to create or delete a
**folder**, or move a note/folder into a different folder. This milestone
closes that gap.

Scope (decided autonomously, recorded in `DECISIONS.md`; reviewed live at the
end):

- **Destination picker, not drag-and-drop.** Moving a note/folder is driven by
  a "Move to…" overlay (same visual family as the quick switcher/palette) that
  lists existing folders for one-click pick. Keyboard- and mobile-friendly,
  and no new DnD machinery to build/test. Drag-and-drop reordering is out.
- **No new context-menu system** *(superseded — see P3)*. Folder create/
  delete originally got the same inline-button treatment notes already have
  in the tree. The milestone review reversed this: too many always-visible
  row buttons. P3 replaces every row button (folder create/move/delete, note
  move, and the pre-existing note delete) with one right-click/long-press
  context menu.
- **Delete folder is destructive and needs confirmation.** Unlike deleting a
  single note (already a one-click "×"), deleting a folder can take an
  unbounded number of notes with it — a confirm step is mandatory.
- **Move reuses the rename plumbing.** `moveNote` (`note.ts`) already handles
  cross-folder moves and prefers the native `FileSystemFileHandle.move()`;
  `renameActive` (`note-controller.ts`) already wraps it with flush + link
  rebase + reactivation. Moving a note generalizes `renameActive` from
  "same folder, new name" to "any destination folder" — no new note-level FS
  primitive needed.

## P1 — Create & delete folder

New `createFolder(dir, path)` (`getDirectoryHandle` per segment,
`create: true`) and `deleteFolder(dir, path)` (`removeEntry(name,
{recursive: true})` on the resolved parent) in `note.ts`, mirroring the
existing `createNote`/`deleteNote` shape. Sidebar tree (`ui.ts`) gets a "+"
button per folder row (and at the root) that prompts for a name and calls
`createFolder`, and a "×" on folder rows that confirms, then calls
`deleteFolder`. The M4 poller and `listNotes`/sweep already ignore folders as
entities (they only ever descend into them), so a freshly-created empty
folder needs no special-casing there — it just won't show notes yet.

**Done =** a new subfolder can be created from any folder row (and the
root) and immediately shows in the tree; deleting a folder asks for
confirmation, then removes it and every note beneath it from disk and from
the tree; deleting an already-gone folder (e.g. a stale render) is a no-op,
not an error.

## P2 — Move a note / move a folder

A "Move to…" overlay (new, palette-styled) listing existing folders (root
included), triggered from a note row and from a folder row. Picking a note's
destination generalizes `renameActive` to accept a target folder + the note's
current name (same flush → `moveNote` → link-rebase → reactivate sequence,
just varying which path segment changes). Picking a folder's destination
calls a new `moveFolder(dir, fromPath, toPath)`: enumerate the subtree
(`listNotes`), `moveNote` each file from its `fromPath`-rooted path to the
equivalent `toPath`-rooted path (rebasing inbound links per moved note, same
as a rename), then remove the emptied source folder tree once every file has
moved. Refuses a folder move into its own subtree (would orphan/self-nest).

**Done =** a note can be moved into any other folder (including root) via the
picker, its content and inbound links survive intact; a folder can be moved
the same way, taking all its notes with it, links updated per note; a folder
cannot be moved into itself or a descendant of itself.

## P3 — Context menu for row actions

Milestone-review correction: P1/P2's always-visible row buttons ("+"/"→"/"×"
on a folder row, "→" on a note row) cluttered the list. Replaces every one of
them — plus the pre-existing note "×" (FEAT-0012, predates this milestone,
swept in for full consistency: zero inline buttons on any row) — with a
single right-click context menu, reachable by mouse (`contextmenu`) and by
touch (long-press, decided live in the review — mobile UX is otherwise its
own future milestone, M17, but this one gesture ships now rather than
leaving touch with no way to reach these actions at all).

A new generic `tree-menu.ts` (decoupled from CodeMirror, unlike the existing
editor `context-menu.ts` it visually mirrors via the shared `.cm-context-menu`
style) opens a positioned, dismiss-on-outside-click/Escape menu of `{label,
run}` items. The actions themselves are unchanged — a menu item's `run()`
calls the exact same `NoteListHandlers` callback a button used to
(`onMoveNote`/`onMoveFolder`/`onCreateFolder`/`onDeleteFolder`/`onDelete`), so
`main.ts`'s confirm/prompt logic for each is untouched; only the trigger
surface moves from a button to a menu item.

**Done =** a note row's context menu offers Move/Delete; a folder row's
offers New subfolder/Move/Delete; each fires the same handler its button
used to; long-press opens the same menu on a touch device; no row shows an
inline action button anymore.
