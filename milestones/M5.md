# M5 — Finish the editor (rendering gaps)

**Goal:** pay down the debt on the heart of the product. M2 promised to hide
markdown markup so text reads as rich content, but several constructs still show
their raw markers — reported from real use. This milestone is **not** a new
feature; it is M2's promise kept. The work lives in `markdown-render.ts` (the
decoration layer) plus `markdown-transforms.ts` (Clear Formatting).

Success: a note containing a fenced code block, a blockquote, and a bulleted list
reads as rendered content — no literal ```` ``` ````, `>`, or `*`/`-` markers
on screen — while the bytes on disk stay exactly the CommonMark the user typed
(the file-fidelity moat). "Clear formatting" / `/clear` removes *all* formatting
it touches, inline code included, not just headings.

## Progress
- [x] Phase 1 — Block-construct rendering
- [ ] Phase 2 — Clear Formatting completeness

Phase 1 extends the existing hide-and-style decoration layer to the three
block-level constructs M2 left raw. Phase 2 closes the matching gap on the
inverse operation: Clear Formatting must strip every mark the renderer hides,
not only the heading prefix.

Constraints that apply throughout:
- **File fidelity is the moat.** The renderer is pure-read: it never mutates the
  document, so the on-disk bytes are untouched. Clear Formatting writes only
  plain CommonMark characters back.
- **No syntax hiding on cursor reveal.** Markup stays hidden on every line,
  including the caret line (the M2 no-flicker rule) — block constructs follow the
  same rule as inline ones.
- **Lean.** Reuse the existing `markdownSyntaxRanges` + decoration plumbing; add
  line decorations only where a span decoration can't express a block.

## Phase 1 — Block-construct rendering
The decoration layer (`markdown-render.ts`) currently handles ATX headings and
inline marks (bold/italic/code). Extend it to the three block constructs still
showing raw markers:

- **Fenced code blocks** (```` ``` ````): collapse the opening fence line (the
  ```` ``` ```` + optional language info + its newline) and the closing fence
  line; style the code body as a monospace block with a subtle background. The
  bytes — fences, language, indentation — stay on disk untouched.
- **Blockquotes** (`> …`): hide every `>` marker (and its following space) on
  each quoted line, including continuation lines; style the quoted content with a
  left border / muted treatment so it reads as a quote.
- **Unordered lists** (`*`, `-`): hide the literal `*`/`-` marker (and its
  trailing space) and render a real bullet glyph in its place; the list item text
  reads as a list item. Ordered lists are out of scope here (no raw-marker
  complaint reported; the `1.` reads fine).

To express block-level styling (a code block's background, a quote's border, a
list's bullet) the layer gains **line decorations** alongside the existing
span/replace decorations — `markdownSyntaxRanges` returns them and the view
plugin applies them. Keep `markdownSyntaxRanges` pure and range-scoped exactly as
today.

- **Done =** unit tests cover the new classifications in `markdownSyntaxRanges`
  (fenced code hides both fences + styles the body; blockquote hides every `>`
  marker incl. continuation + styles the quote; bullet list hides the marker +
  emits a bullet/line decoration) with the FSA-independent parser, and the live
  editor shows a code block, a quote, and a bulleted list with no raw markers
  while the file on disk is byte-identical to what was typed.

## Phase 2 — Clear Formatting completeness
"Clear formatting" (right-click menu) and `/clear` (slash command) today only
reset the heading level — they leave bold, italic, and inline code in place. M2's
promise is that the user never has to touch raw markup; Clear Formatting must
honor that by stripping **everything** it can, across the whole selection:

- Unwrap inline marks — **bold**, **italic**, **inline code** — to their inner
  text (reuse the parsed-structure `unwrap` from `toggleInline`, never a raw
  character scan, to avoid the `*`-inside-`**` ambiguity).
- Strip block prefixes — heading `#`, blockquote `>`, list `*`/`-` markers — from
  every line the selection touches, leaving plain paragraph text.
- Audit the renderer's hide set while doing this: whatever the renderer hides,
  Clear Formatting should be able to remove. Anything deliberately left (e.g.
  ordered-list numbers, if lists land that way) is documented, not silent.
- One transform, reused by both the menu and the slash command (the FEAT-0007/8/9
  single-definition rule). Returns `null` when nothing would change so it never
  pollutes the undo history.

- **Done =** unit tests show Clear Formatting on a selection containing a heading,
  bold, italic, inline code, a quote, and a bullet leaves plain prose with no
  markers; the right-click "Clear formatting" item and `/clear` both run it; a
  no-op selection issues no transaction.

## Out of scope (deliberately)
- **Ordered lists, task lists, tables, images, links** — no raw-marker complaint
  reported; links are M8. This milestone closes only the three reported gaps.
- **Nested / multi-level lists** — render the flat case; deep nesting is not a
  reported pain and can wait.
- **Syntax highlighting inside fenced code** — the body renders as a plain
  monospace block; per-language coloring is polish, not a missing promise.
