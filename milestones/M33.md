# M33 — Multiple vaults / workspaces

**Goal:** make working across several folders painless. Promotes the backlog
"Workspaces" item, pulled forward by real friction. Two parts:
- **Switch between granted folders** — keep the set of previously-granted folder
  handles (already candidates for IndexedDB) and offer quick switching between them,
  instead of re-picking via the native picker each time.
- **Per-window vault identity** — two windows open on two vaults currently share one
  origin-global handle in IndexedDB, so a refresh can swap a window to "the other"
  vault. Give each window a stable vault identity (a `?ws=` URL param) so a reload
  re-attaches to the *same* vault and the two windows stay independent.

Storage/UI only — **no change to note file bytes** (the moat is untouched): the
folder handles live in IndexedDB (browser-private), never in the user's folders.

Scope (decided autonomously, recorded in `DECISIONS.md`; reviewed live at the end):

- **Vault model** — replace the single `brulion:dir` handle with a **set** of vaults,
  each `{ id, handle, name }`; `id` is a short opaque generated key (stable, not the
  folder name). A legacy single handle migrates in as the first vault.
- **Per-window identity via `?ws=<id>`** — the window's vault id rides the URL query;
  with the existing `#/path` note hash (FEAT-0036), the URL fully describes a window's
  state (which vault + which note), so reload re-attaches deterministically and two
  windows are independent without any per-window IndexedDB state.
- **Session state stays global for now** — recency / expanded folders / sidebar
  width+collapse remain origin-global (a small imperfection across differing vaults);
  per-vault session state is a possible follow-up, not in this milestone.
- **UI says "folder"** (Open folder / Switch folder); "vault" is the code/docs term.

## P1 — Vault model + per-window identity

A new `src/vaults.ts` over IndexedDB: a list of `{ id, handle, name }` with
`listVaults` / `addVault` (dedup by `isSameEntry`, generate id) / `getVault(id)` /
`removeVault(id)`, plus a one-time migration of the legacy `brulion:dir` handle into
the set. The open/restore flow (`ui.ts` `openFolder`/`restoreFolder`, wired in
`main.ts`) reworks to: opening a folder adds it to the set and stamps `?ws=<id>` on
the URL (replaceState); on load the window reads `?ws` and re-attaches to *that*
vault (re-granting permission if needed), falling back to the migrated/last vault or
the welcome screen when there's none or no `?ws`. No switcher UI yet (that's P2);
adding a vault is still the native picker.

**Done =** opening a folder records it in the vault set and gives the window a
`?ws=<id>`; a reload re-attaches to the same vault (not "whichever was last opened
in any window"); two windows on different vaults each reload to their own vault; a
pre-M33 single-folder user is migrated transparently; the on-disk note bytes are
unchanged.

## P2 — Vault switcher (palette action + chooser)

A command-palette action **"Switch folder…"** (reframing the existing one) plus/or a
new **"Open another folder…"**, and a chooser overlay (same pattern as the quick
switcher) listing the granted vaults by name for one-click switching — re-granting
permission on selection if the handle lost it. A settings surface to forget a vault
(remove from the set). Switching points the window at the chosen vault and updates
`?ws`. Pure UI/UX over the P1 model.

**Done =** a palette action opens a chooser of the granted folders; picking one
switches the window to it (re-granting permission if needed) and updates `?ws`;
the user can forget a vault; switching writes no note bytes.
