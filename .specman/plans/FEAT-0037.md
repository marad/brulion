## Design

New module `src/link-complete.ts` — a `@codemirror/autocomplete` source for
wikilinks, peer to `slash-commands.ts`:

- `wikilinkSource(context: CompletionContext): CompletionResult | null`
  - `context.matchBefore(/\[\[[^\]\n]*/)` — matches `[[` + partial target (no `]`,
    no newline). Null → return null.
  - `from = match.from + 2` (right after `[[`); `query = match.text.slice(2)`.
  - paths = `[...context.state.facet(linkContext).notePaths]` (one source of truth).
  - `searchNotes(query, paths).matches` → ranked note paths (ignore `.create`).
  - Empty matches → return null (nothing to suggest).
  - options: `{ label: displayName(path), type: "text", apply: applyWikilink(path) }`.
  - Return `{ from, options, filter: false }` (no `validFor`) so CodeMirror keeps our
    `note-search` order authoritative and re-queries the source per keystroke for a
    fresh ranking.
- `applyWikilink(path)` → `(view, _completion, from, to) => void`
  - `insert = displayName(path)`; reuse a trailing `]]` if `doc.sliceString(to, to+2)
    === "]]"`, else append `]]`.
  - Dispatch `changes:{from,to,insert}`, caret `selection:{anchor: from + insert.length + 2}`
    (after the `]]`, same formula whether reused or appended).
- `export const wikilinkCompletions = markdownLanguage.data.of({ autocomplete: wikilinkSource })`.

Wire in `editor.ts`: add `wikilinkCompletions` to the extensions array next to
`slashCommands`. No other file changes (the `linkContext` facet is already populated
by `setLinkContext` in `main.ts` on every active-note/list change).

## Tests

Unit tests — `src/link-complete.test.ts` (vitest + happy-dom), state configured
with `markdown()` + `linkContext.of({activeNote, notePaths})`:

- **AC-1** — `wikilinkSource` at caret after `[[` returns a result whose options are
  all notes (empty query) and contains no create row.
- **AC-2** — after `[[dia`, options are ranked by `searchNotes` order (assert order
  equals `searchNotes("dia", paths).matches.map(displayName)`); `from` is right after
  `[[`; `filter:false` is set.
- **AC-3** — `apply` on a buffer `[[dia` (caret at end): result text is
  `[[<displayPath>]]` and the caret lands after `]]` (build an `EditorView`, run the
  option's `apply`, assert `doc` and `selection.main.head`).
- **AC-4** — `apply` on `[[dia]]` (caret before `]]`): result is a single
  `[[<displayPath>]]`, no doubled brackets.
- **AC-5** — `wikilinkSource` returns null (or empty→null) for a no-match query; the
  source never yields a create row. (Dangling-link allowance is structural: the
  source only suggests; it inserts/creates nothing.)
- **AC-6** — with two notes sharing a basename in different folders, the inserted
  target is the full display path (covered by AC-3 using a nested note path).
- **AC-7** — `slashSource` unchanged (existing `slash-commands.test.ts` stays green);
  `wikilinkSource` returns null for a `/` slash context (no `[[`).
- **AC-8** — read-only is structural: the source/apply only read the facet and edit
  the buffer; asserted by inspection (no fs import in `link-complete.ts`).

E2E — `e2e/link-autocomplete.spec.ts` (Playwright, OPFS-backed FSA): seed a couple
notes, open one, type `[[` then a few chars, accept with Enter, assert the buffer
contains a well-formed `[[path]]` and (via rendering) it resolves to the note (not
broken). Covers AC-1/2/3 end-to-end in real Chromium.

Build/verify: `npm run build` (tsc + vitest) and `npm run e2e` (the wikilink/slash
suites must stay green — no regression).
