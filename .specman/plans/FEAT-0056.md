# Sync plan — FEAT-0056 mermaid diagram rendering in fenced blocks

Started: 2026-06-24T15:29:05.519Z
Snapshot state: new
Drift summary: 0 added (whole spec)

Implemented via `/excavate` (new module). The module design (diagram + signatures)
lives in `docs/excavation/`; this plan maps the ACs to the resulting code.

## AC-1 (added) — A `mermaid` fenced block renders as a diagram

Approach: New `src/mermaid-render.ts` exporting `mermaidRendering: Extension`, a
`StateField` that scans the Lezer tree for **closed** `FencedCode` nodes whose info
string is `mermaid` (mirroring the existing fenced-code detection in
`markdown-render.ts`), and for each block not overlapping the selection emits
`Decoration.replace({ widget: new MermaidWidget(source), block: true })` over the
block range. Registered in `src/editor.ts` after `markdownRendering`. `MermaidWidget`
renders the diagram into its DOM.

Files: src/mermaid-render.ts, src/editor.ts, src/styles.css

## AC-2 (added) — Source bytes are never modified

Approach: Pure decoration — the field only adds `Decoration.replace` over existing
ranges and never dispatches document changes. No serialize path. Verified by an e2e
reading the file/doc after render + edit.

Files: (none — property of the decoration-only design)

## AC-3 (added) — Selecting inside the block reveals the editable source

Approach: In the field's decoration build, skip (emit no replace for) any mermaid
block whose range overlaps the current selection — leaving the raw fenced source
visible/editable. Rebuild the field on `docChanged || selectionSet` so moving out
re-renders from the current source. Same selection-overlap reveal pattern as
`markdown-render.ts`.

Files: src/mermaid-render.ts

## AC-4 (added) — An unrelated edit does not re-run Mermaid for an unchanged block

Approach: `MermaidWidget.eq(other)` compares the diagram source string. On a rebuild
triggered by an unrelated edit, CodeMirror sees the widget as equal and reuses the
existing DOM, so `toDOM()`/the async render does not run again.

Files: src/mermaid-render.ts

## AC-5 (added) — An invalid diagram shows an in-place error, not a crash

Approach: The async render path wraps `mermaid.render`/`parse` in try/catch; on
failure it fills the widget DOM with a discreet error box containing the message,
rather than throwing. The field build and other decorations are unaffected.

Files: src/mermaid-render.ts, src/styles.css

## AC-6 (added) — The Mermaid engine is loaded lazily

Approach: A dedicated `src/mermaid-engine.ts` wraps `import("mermaid")` +
one-time `initialize({ startOnLoad: false })` behind a module-level singleton
promise (`loadMermaid()`); `mermaid-render.ts` calls it only inside the widget's
async render. Vite splits `mermaid-engine` (and Mermaid) into its own chunk, absent
from the main bundle. Verified by an e2e that intercepts network requests: opening a
plain note requests no mermaid chunk; opening a note with a Mermaid block does.

Files: src/mermaid-engine.ts, src/mermaid-render.ts

## Tests

- src/mermaid-render.test.ts (unit, happy-dom; Mermaid NOT invoked) — block
  detection finds closed `mermaid` fences and their ranges, ignores a non-`mermaid`
  info string and an unterminated fence; the selection-overlap filter reveals a block
  whose range intersects the selection and renders one that doesn't; `MermaidWidget.eq`
  is true for equal source and false for changed source.
- e2e/mermaid.spec.ts (Chromium) — a valid block renders an `svg` (AC-1); the doc
  text is unchanged after render (AC-2); clicking into the block shows the raw source
  and moving out re-renders (AC-3); an invalid block shows the error box, editor still
  works (AC-5); network interception confirms lazy load (AC-6).
