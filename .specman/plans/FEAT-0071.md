## Implementation

Whole-spec plan (brand-new, same convention as FEAT-0068/0069/0070's plans).

### 1. `src/tree-menu.ts` — new file, mirrors `context-menu.ts`'s `openMenu`

`context-menu.ts`'s `openMenu(view, x, y, items)` (lines 41-97) is the exact
shape needed — position at `(x, y)`, clamp into the viewport, dismiss on
outside `pointerdown` (capture) or Escape, single-menu-at-a-time via a
module-level `close` — but it's typed around `EditorView`/`MenuItem.run(state)
→ TransactionSpec`, which the tree has no use for. `tree-menu.ts` is the same
shape with the CodeMirror coupling stripped:

- `export interface TreeMenuItem { label: string; run: () => void }`
- `export function openTreeMenu(x: number, y: number, items: TreeMenuItem[]): void`
  — builds `.cm-context-menu` (reusing the **existing** class/CSS as-is; no
  new styling — the look isn't CodeMirror-specific, just visually shared), one
  `button[role=menuitem]` per item, appended to `document.body`; a click runs
  the item then closes. Viewport-clamp and outside-click/Escape dismissal
  copied verbatim from `context-menu.ts`'s logic (no `view.focus()` call —
  nothing to refocus in the tree).
- `export function closeTreeMenu(): void` — exported for tests/cleanup
  (`context-menu.ts` doesn't export its `closeMenu`, but the tree menu isn't
  torn down by a CodeMirror `ViewPlugin.destroy`, so callers need it directly).

### 2. `src/long-press.ts` — new file

- `export function wireLongPress(el: HTMLElement, onLongPress: (x: number, y: number) => void, ms = 500): void`
- `touchstart` (single touch only — cancel on a second) starts a `setTimeout`;
  `touchmove` past a small tolerance (10px) or `touchend`/`touchcancel`
  cancels it; firing calls `onLongPress(startX, startY)`. `touchend` calls
  `event.preventDefault()` **only when the press actually fired**, to
  suppress the synthetic mouse events (and thus the row's own click handler)
  a touch browser dispatches after `touchend` — an early lift behaves as an
  ordinary tap (AC-6).
- Unit-testable in happy-dom by dispatching plain `Event("touchstart"/"touchmove"/"touchend")`
  instances with a `touches` array attached via `Object.defineProperty` (no
  native `TouchEvent` needed) and `vi.useFakeTimers()` to control the
  threshold deterministically.

### 3. `src/ui.ts` — remove every row button, wire the menu + long-press

- Delete `rowActionButton` (dead code once its 5 call sites are gone) and the
  `.folder-row` wrapper div entirely — a folder node reverts to exactly its
  pre-P1 shape (`folder` div containing `header` + `children` directly, no
  intermediate row wrapper needed once there are no sibling buttons to lay
  out next to the header).
- `renderNode`'s folder branch: after building `header`/`children` (unchanged
  toggle logic), attach `header.addEventListener("contextmenu", ...)` and
  `wireLongPress(header, ...)`, both opening
  `openTreeMenu(x, y, [{label:"New subfolder…", run: () => handlers.onCreateFolder?.(node.path)}, {label:"Move…", run: () => handlers.onMoveFolder?.(node.path)}, {label:"Delete", run: () => handlers.onDeleteFolder?.(node.path)}])`.
  `folder.append(header, children)` (no `row` wrapper).
- `renderNoteRow`: back to just `row` (`.note-row`) containing `nameButton`
  only (no `.note-move`/`.note-delete` buttons). `row.addEventListener("contextmenu", ...)`
  and `wireLongPress(row, ...)` open
  `openTreeMenu(x, y, [{label:"Move…", run: () => handlers.onMoveNote?.(node.path)}, {label:"Delete", run: () => handlers.onDelete(node.path)}])`.
- `NoteListHandlers` interface **unchanged** — every callback it already
  exposed is reused verbatim as a menu item's `run()`; `main.ts`'s
  prompt/confirm logic for each is untouched.
- Both `contextmenu` listeners call `event.preventDefault()` first (so the
  browser's native menu never shows over a row).

### 4. `src/styles.css` — remove the now-dead row-button rules

Delete `.note-move`/`.note-delete`, `.folder-row`/`.folder-create`/
`.folder-move`/`.folder-delete` rules (P1/P2 additions) — nothing references
those classes anymore. `.cm-context-menu` (reused, not duplicated) needs no
change.

### 5. Existing test/e2e fallout — update selectors, not behavior

Every assertion that clicked `.note-delete`/`.note-move`/`.folder-create`/
`.folder-move`/`.folder-delete` now instead right-clicks the row/header and
clicks the matching `.cm-context-menu button[role=menuitem]` by its label
text. Affected: `src/ui.test.ts` (the FEAT-0069/0070 row-button describe
blocks get rewritten against the menu instead of deleted — same behavior,
new trigger), `e2e/note-crud.spec.ts`, `e2e/subfolders.spec.ts`,
`e2e/folder-crud.spec.ts`, `e2e/move-note.spec.ts`, `e2e/move-folder.spec.ts`.
None of these change what they assert about the underlying action (confirm
dialogs, disk state, link rewriting) — only how the action is triggered in
the test.
