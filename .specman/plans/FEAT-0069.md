# Sync plan — FEAT-0069 Create and delete folders

## Implementation

A brand-new spec; treated as one whole-spec plan rather than per-AC sections
(same convention as FEAT-0068's plan). The core wrinkle found while grounding
this in the real code: `buildNoteTree` (`src/ui.ts:101`) has always derived
folder nodes purely as a side effect of note paths — an empty folder has never
had a representation. AC-1/AC-2 require a freshly created *empty* folder to
show up in the tree, so the folder listing needs to see real directories, not
just infer them from `.md` paths. That drives most of the shape below.

### 1. `src/note.ts` — storage layer

- **`listFolders(dir, maxConcurrent?, signal?): Promise<string[]>`** — a sibling
  of `listNotes`/`collect` (`note.ts:136-194`), its own small recursive walk
  (same semaphore-bounded, abortable shape) that records each **directory's**
  relative path (`prefix + entry.name`, no trailing slash) instead of `.md`
  files, excluding the picked root itself. Sorted with the same
  `localeCompare(..., { sensitivity: "base" })` as `listNotes`. Kept as an
  independent walk (not threaded into `listNotes`'s return shape) so the
  ~6 existing `listNotes` call sites (`note-controller.ts`, `settings.ts`,
  `ui.ts`, tests) stay untouched — the cost is a second walk when the sidebar
  needs both, acceptable at this project's scale.
- **`CreateFolderResult = { status: "created" } | { status: "exists" }`**
  and **`createFolder(dir, path): Promise<CreateFolderResult>`** — mirrors
  `createNote`/`getExisting` (`note.ts:260-284`) exactly, swapping
  `getFileHandle` for `getDirectoryHandle`: a new private `getExistingFolder`
  (parallel to `getExisting`, `note.ts:354-370`) checks via
  `parent.getDirectoryHandle(file)` (no `create`); if found, `exists`; else
  `resolveParent(dir, folders, true)` then `parent.getDirectoryHandle(file,
  { create: true })`. `isAbsent` (a file blocking a folder segment, or
  occupying the target name) collapses to `exists`, reusing the existing
  helper unchanged.
- **`deleteFolder(dir, path): Promise<void>`** — mirrors `deleteNote`
  (`note.ts:288-300`) exactly, swapping `removeEntry(file)` for
  `removeEntry(file, { recursive: true })`. Idempotent on an already-missing
  folder (`isNotFound`, reused as-is).

### 2. `src/note-name.ts` — folder-path validation

- Extract the existing per-segment validator out of `normalizeNoteName`
  (`note-name.ts:36-58`, the loop body at lines 43-53) into a private
  `validateSegments(raw: string[]): { ok: true; segments: string[] } | {
  ok: false; reason: string }`. `normalizeNoteName` calls it, then keeps its
  own last-segment `.md` handling (line 45, 56) on top.
- **`normalizeFolderPath(input): { ok: true; path: string } | { ok: false;
  reason: string }`** — trims, splits on `/`, calls `validateSegments`, joins
  with no `.md` suffix anywhere (every segment is a folder name). Same
  empty/`.`/`..`/unsafe-character rejections as notes, by construction (one
  validator, two thin callers).

### 3. `src/note-controller.ts` — controller operations

- **`addFolder(path: string): Promise<AddNoteResult>`** — same shape as
  `addNote` (`note-controller.ts:610-626`): `abortPendingRelist()` →
  `serialize<AddNoteResult>` → guard `!dir` / `conflict` →
  `normalizeFolderPath` → `createFolder(dir, normalized.path)` → on `exists`
  return `{ ok: false, reason: "A folder already exists there." }` → else
  `opts.onListChanged?.(notes, activeName)` (notes are unchanged — an empty
  folder adds no `.md` file — this call exists purely to trigger a re-render;
  the handler re-fetches `listFolders` itself, see §4) → `{ ok: true }`. No
  flush/activate needed — creating a folder never touches the active note.
- **`removeFolder(path: string): Promise<void>`** — same shape as `removeNote`
  (`note-controller.ts:627-648`): `abortPendingRelist()` →
  `serialize(async () => ...)` → guard `!dir || conflict` → if the active note
  is inside the folder (`activeName === path || activeName.startsWith(path +
  "/")`), drop its dirty flag and `flushAndWait()` first (mirrors the
  active-note-deletion race guard) → `deleteFolder(dir, path)` → drop any
  cached content for notes that were under `path` (`contentCache` keys
  starting with `path + "/"`) → `notes = await listNotes(dir)` → if the active
  note was inside the deleted folder, `await activate(dir,
  pickActiveNote(notes, null))` (the exact FEAT-0012 AC-7 fallback: `start.md`
  → first remaining note → empty `start` buffer) → else
  `opts.onListChanged?.(notes, activeName)`.

### 4. `src/ui.ts` — tree rendering + row actions

- **`buildNoteTree(paths: string[], folders: string[] = [])`** — reworked to
  merge `paths` and `folders` into one sorted list (`[...paths,
  ...folders].sort(...)`, same `localeCompare` as today) and walk it once via
  a shared `ensureChain(segments)` helper (interns/creates folder nodes along
  a path, factored out of the current inline loop body at `ui.ts:105-120`): a
  folder-path item just materializes its chain (no leaf pushed); a note-path
  item materializes its parent chain then pushes the `NoteLeaf`. Merging and
  re-sorting (rather than two passes) keeps empty folders correctly
  interleaved alphabetically with their populated siblings instead of always
  trailing after them. Existing single-arg call sites (`ui.test.ts`) keep
  working unchanged (`folders` defaults to `[]`).
- **`NoteListHandlers`** (`ui.ts:125-130`) gains `onCreateFolder: (parentPath:
  string) => void` and `onDeleteFolder: (path: string) => void`.
- **`renderNoteList`** (`ui.ts:144-155`) gains a `folders: string[] = []`
  param, passed through to `buildNoteTree`.
- **`renderNode`**'s folder branch (`ui.ts:165-197`): wrap the existing
  `header` button (untouched — same class, same toggle-click listener, so
  `.folder-header` e2e locators in `subfolders.spec.ts` need no changes) in a
  new `.folder-row` div alongside two sibling buttons, mirroring
  `renderNoteRow`'s `.note-name` + `.note-delete` pattern (`ui.ts:200-226`):
  - `.folder-create` — text glyph `+`, `title`/`aria-label` `` `New folder in
    ${node.name}` ``, click → `handlers.onCreateFolder(node.path)`.
  - `.folder-delete` — text glyph `×`, `title`/`aria-label` `` `Delete
    ${node.name}` ``, click → `handlers.onDeleteFolder(node.path)`.
  Plain text glyphs (not Lucide icons) to match `.note-delete`'s existing
  local convention for row-level actions, not the header's M27 icon set.

### 5. `index.html` + `src/main.ts` — wiring

- `index.html`: a new `#sidebar-new-folder` button (plain text, `+ New
  folder`, matching `#sidebar-search`'s plain-text style) as a sibling
  between `#sidebar-search` and `#note-list` (`index.html:91-96`).
- `src/main.ts`: extend the shared `noteListHandlers` object
  (`main.ts:370-390`) with:
  - `onCreateFolder: (parentPath) => { const name = window.prompt("New folder
    name:"); if (!name) return; const path = parentPath ? `${parentPath}/${name}`
    : name; void controller.addFolder(path).then(r => { if (!r.ok)
    window.alert(r.reason) }) }` — reuses the native `window.prompt`/`alert`
    pattern already established for note delete's `window.confirm`
    (`main.ts:379`); no new modal component.
  - `onDeleteFolder: (path) => { if (window.confirm(`Delete "${path}" and
    everything inside it? This removes every note beneath it from your
    folder.`)) { void controller.removeFolder(path) } }`.
  - `#sidebar-new-folder`'s click listener calls the same `onCreateFolder`
    logic with `parentPath = ""` (root).
- Both `renderNoteList` call sites (cache paint ~`main.ts:422`, real
  `onListChanged` ~`main.ts:479`) pass a `folders` array: the cache-paint call
  passes `[]` (empty folders are a rare, brand-new concept — no cache to seed
  from, consistent with "the cache is a hint, never authoritative"); the real
  `onListChanged` handler calls `await listFolders(dir)` before
  `renderNoteList`, so every real re-render (including ones the M4 poller
  triggers via the same callback) reflects current empty folders too.
- `styles.css`: a `.folder-row` flex rule (header grows, the two action
  buttons sized like `.note-delete`) — same pattern `.note-row` already uses.
