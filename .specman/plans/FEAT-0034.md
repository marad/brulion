# Sync plan — FEAT-0034 Move a note on disk and rename the active note

Started: 2026-06-22T19:50:59.714Z
Snapshot state: new
Drift summary: 0 added (whole spec)

## Approach

Two units, no new module — a storage function plus a controller method.

### Storage: `moveNote` (AC-1..AC-5) — `src/note.ts`

NEW CODE. Add `export type MoveResult = { status: "moved" } | { status: "exists" } | { status: "missing" }`
and `export async function moveNote(dir, from, to): Promise<MoveResult>`.

- `from === to` → `{ status: "moved" }` immediately (no-op, AC-5).
- Resolve the source via the existing private `getExisting(dir, from)`; `null` →
  `{ status: "missing" }` (AC-4).
- Guard the destination: `getExisting(dir, to)` non-null → `{ status: "exists" }`,
  source untouched (AC-3, the moat — never clobber).
- Resolve the destination parent with the existing private
  `resolveParent(dir, splitPath(to).folders, true)` to materialize folders
  (AC-2), then `await sourceHandle.move(destParent, destFile)` — the native FSA
  primitive, preserving bytes (AC-1, AC-2). Return `{ status: "moved" }`.

Reuses the existing private `splitPath`/`resolveParent`/`getExisting` helpers —
no duplication. `FileSystemFileHandle.move` is typed via a local cast (the lib
DOM types may not yet include it); narrow the cast to the two-arg
`move(dir, name)` form.

### Controller: `renameActive` (AC-6..AC-11) — `src/note-controller.ts`

NEW CODE. Add `renameActive(name: string): Promise<AddNoteResult>` to the
`NoteController` interface and implement it inside `createNoteController`, run
through the existing `serialize(...)` slot so it cannot interleave with a switch,
save, or poll.

Body:
1. `if (!dir) return { ok: false, reason: "Open a folder first." }`
2. `if (conflict) return { ok: false, reason: "Resolve the conflict first." }` (AC-10)
3. `const normalized = normalizeNoteName(name)`; if `!normalized.ok` return its
   reason (AC-9).
4. `if (normalized.filename === activeName) return { ok: true }` — no-op rename
   (AC-11).
5. `await flushAndWait()` — flush pending edits so the moved file carries them
   (AC-7). Re-check `conflict` after the flush (a flush can raise one); if now
   conflicted, return the resolve-first reason.
6. `const result = await moveNote(dir, activeName, normalized.filename)`.
   - `missing` → return `{ ok: false, reason: "The note no longer exists." }`.
   - `exists` → return `{ ok: false, reason: "A note with that name already exists." }` (AC-8).
7. On `moved`: refresh `notes = await listNotes(dir)`, then
   `await activate(dir, normalized.filename)`. `activate` → `load` re-reads the
   file (adopting the new `lastModified`), persists it as the active note, and
   fires `onListChanged` so the sidebar/link-context/header follow (AC-6).
   Return `{ ok: true }`.

`load` resets `dirty`/`lastModified` from the moved file, so the next save bases
off the right mtime. The move happening inside `serialize` means the next poll
already sees `activeName` at the new path — no spurious external delete/create.
