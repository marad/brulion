### Pure: `frontmatterRange` (AC-1..AC-3) â€” `src/frontmatter.ts`

NEW export `frontmatterRange(state: EditorState): { from: number; to: number } | null`.
Operates on `state.doc` (line access). Rule:
- Line 1's text, trimmed of trailing `\r`/spaces, must equal `---` (AC-3: else
  `null`). An empty doc or a first line that isn't `---` â†’ `null`.
- Scan lines 2..N for the first whose trimmed text is exactly `---` or `...`
  (the closing delimiter). None found â†’ `null` (AC-2: unclosed stays raw).
- Return `{ from: line1.from, to: closingLine.to }` (AC-1).

Pure: reads doc only, mutates nothing. Unit-tested directly over `EditorState`.

### Exclude the frontmatter region from markdown rendering â€” `src/markdown-render.ts`

The inline `renderPlugin` and the `blockRenderingField` don't know about
frontmatter, so in the **expanded** state they'd hide/style markdown-active
characters inside the raw YAML (a `# comment` line â†’ heading, `**x**` â†’ bold) â€”
breaking the "raw, opaque" promise. Fix at the two build sites: compute
`frontmatterRange(state)` once and drop any `hidden`/`mark`/`line`/`bullet` whose
span intersects `[fm.from, fm.to)`. A small `intersects(a, b, lo, hi)` helper.
One-way dependency `markdown-render â†’ frontmatter` (frontmatter imports neither).
Correct in both states (collapsed: the region is block-replaced anyway; expanded:
raw text shows unstyled).
