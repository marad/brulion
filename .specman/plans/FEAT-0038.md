## Design

Pure-logic rewrite of `fuzzyScore` in `src/note-search.ts`. `searchNotes` is
untouched (it already sorts by `score desc, path asc` and computes `create`); only
the number `fuzzyScore` returns changes, so every caller (`quick-switcher.ts`,
`link-complete.ts`) keeps working unchanged.

Two scoring tiers, with a hard band gap so tier 1 always beats tier 2:

- **`isSeparator(ch)`** ā€” unchanged: `" " | "/" | "-" | "_"`.
- **`isBoundary(t, i)`** ā€” `i === 0 || isSeparator(t[i - 1])`. A match position that
  begins a segment/word.

**Tier 1 ā€” literal contiguous substring (the winner band).** If the lowercased
query is a substring of the lowercased target, scan every start index `p` where
`t.startsWith(q, p)` and score:

    SUBSTRING_BASE + (isBoundary(t, p) ? BOUNDARY : 0)

Take the max over all `p` (so a segment-start occurrence beats a mid-token one ā€”
AC-4) and return it. `SUBSTRING_BASE` is set far above any reachable tier-2 score
(see below) so any substring match outranks any gapped match ā€” AC-1.

**Tier 2 ā€” best-alignment subsequence (the fallback band).** When the query is only
a subsequence, compute the **maximum** score over all alignments via a small DP
(not greedy ā€” AC-3). `f[i][j]` = best score matching `q[0..=i]` with `q[i]` placed
at `t[j]` (only where `t[j] === q[i]`):

    f[0][j] = CHAR + (isBoundary(t,j) ? BOUNDARY : 0)
    f[i][j] = CHAR + (isBoundary(t,j) ? BOUNDARY : 0)
              + max(
                  f[i-1][j-1] + CONTIG,                       // contiguous: no gap penalty
                  best_{j' <= j-1}( f[i-1][j'] - GAP*(j-j'-1) ) // gapped: interior gap only
                )

The gapped term is computed in O(1) with a running prefix-max of `f[i-1][j'] +
GAP*j'` (since `-GAP*(j-j'-1) = (GAP*j') - GAP*(j-1)`), so the whole pass is
O(|q|Ā·|t|) ā€” AC bounded-cost. **No leading-distance term** anywhere: the first
matched char pays nothing for how deep it sits, so folder depth is free ā€” AC-2.
Result = `max_j f[|q|-1][j]`, or `null` if any query char never matches (not a
subsequence) ā€” AC-5.

**Constants & the band gap.** `CHAR=10`, `BOUNDARY=10`, `CONTIG=5`, `GAP=1`. Tier-2
max for a length-L query is bounded by `L*(CHAR+BOUNDARY+CONTIG) = 25L`; note paths
are short, but to keep the band guarantee robust set `SUBSTRING_BASE = 100000`
(ā‰« 25L for any realistic L) so tier 1 is always strictly above tier 2 ā€” AC-1. Empty
query returns `0` before either tier (unchanged) ā€” AC-5/AC-6.

This is a single-file change (`src/note-search.ts`); `isSeparator` stays, the
`fuzzyScore` body is replaced, `searchNotes` is unchanged.

## Tests

Unit tests ā€” extend `src/note-search.test.ts` (vitest). Keep the existing
`fuzzyScore`/`searchNotes` cases green (they assert orderings and the
null/empty/contract behavior ā€” all still hold), and add:

- **AC-1** ā€” `searchNotes("0622", ["a/b/c/2026-06-22.md", "06x2x2.md"])` (or
  similar): the contiguous-substring note ranks first even though its path is
  deeper. Assert `matches[0]` is the deep contiguous one. (Use a query that is a
  literal substring of the deep path and only scattered in the shallow one.)
- **AC-2** ā€” `fuzzyScore("notes", "notes.md"-display)` vs the same name nested:
  `fuzzyScore("week", "Week")` equals `fuzzyScore("week", "a/b/c/Week")` ā€” depth
  adds nothing. Assert strict equality of the two numbers.
- **AC-3** ā€” best alignment: a target where chars appear early-scattered and
  late-contiguous, e.g. `fuzzyScore("abc", "a-b-c-abc")`. Assert it equals the
  pure-substring score of `"abc"` in the same string (tier 1), proving the
  contiguous run is found, not the greedy early scatter.
- **AC-4** ā€” segment-start vs mid-token substring: `searchNotes("bar", ["foobar.md",
  "x/bar.md"])` ā†’ `x/bar.md` first (its `bar` starts a segment; `foobar`'s is
  mid-token).
- **AC-5** ā€” purity/contract: `fuzzyScore("xyz", "abc")` is `null`; `fuzzyScore("",
  "anything")` is `0`; case-insensitive still holds (existing tests).
- **AC-6** ā€” `searchNotes` contract: existing empty-query/path-order,
  whitespace-as-empty, create-offer, and existing-note cases stay green unchanged.

No e2e is required (pure logic), but run the switcher/autocomplete e2e suites as a
regression guard since they consume this ranking.
