# Sync plan — FEAT-0079 Name-keyed vault resolution for cross-device permalinks

New spec (whole-spec sync). Small change across three existing modules
(`settings.ts`, `vaults.ts`, `main.ts`) plus tests — no new module → implemented
via **chisel**. Tests before bodies.

## AC-1, AC-2 — Effective workspace name (folder-name default, explicit override)

Approach: Add a `workspace: string` field to the `Settings` model
(`src/settings.ts`): add it to the `Settings` interface, to `DEFAULT_SETTINGS`
(`""`), and to `normalizeSettings` (`typeof r.workspace === "string" ? r.workspace
: ""`, mirroring `journalPath`). Add a pure helper `effectiveVaultName(vault:
Pick<Vault, "name" | "workspace">): string` in `src/vaults.ts` that returns the
trimmed configured `workspace` when non-empty, else the folder `name`. AC-1 (unset
/ blank → folder name) and AC-2 (non-empty → that name) are the two branches of
this helper.

Files: src/settings.ts, src/vaults.ts, src/settings.test.ts, src/vaults.test.ts

## AC-3 — Attach caches the configured workspace name on the vault record

Approach: Add an optional cached `workspace?: string` to the `Vault` interface
(`src/vaults.ts`) and a `setVaultWorkspace(id, workspace)` writer that does a
single idb-keyval `update` read-modify-write (like `touchVault`), storing
`workspace.trim() || undefined` on the matching vault (absent when blank). In
`src/main.ts` `attachVaultNow`, after `openNote(vault.handle)` succeeds (which
loads `.brulion.json` into `currentSettings`), call `setVaultWorkspace(vault.id,
currentSettings.workspace)` so the cache reflects the on-disk name for later
startup resolution.

Files: src/vaults.ts, src/main.ts, src/vaults.test.ts

## AC-4, AC-5, AC-6 — Name-first, id-fallback `?ws` resolution (collision → most-recent)

Approach: Add `resolveVaultRef(ref: string): Promise<Vault | undefined>` in
`src/vaults.ts`: list vaults once (already most-recent-first), return the first
whose `effectiveVaultName` equals `ref` (so a collision yields the most-recent —
AC-6), else the first whose `id` equals `ref` (legacy fallback — AC-5), else
`undefined`. In `src/main.ts` `resolveStartupVault`, replace the `getVault(ws)`
call with `resolveVaultRef(ws)`. AC-4 is the single-name-match path.

Files: src/vaults.ts, src/main.ts, src/vaults.test.ts

## AC-7 — Unknown `?ws` falls back and preserves the note hash

Approach: Already the shape of `resolveStartupVault` — on `resolveVaultRef`
returning `undefined` it falls through to `listVaults()[0]` (most-recent) or
`undefined` (welcome). The `#/…` hash is never touched by vault resolution
(`stampWorkspace`/`unstampWorkspace` preserve `location.hash`; the note route owns
the hash, FEAT-0036), so it survives to be resolved once a folder is granted.
Covered by a `resolveVaultRef` miss test plus asserting the hash is untouched.

Files: src/vaults.ts (resolver), src/main.ts (no new code beyond the AC-4 swap)

## AC-8 — The stamped `?ws` is the effective name, not the opaque id

Approach: In `src/main.ts` `attachVaultNow`, stamp the effective name instead of
`vault.id`. Stamp early (before `openNote`) with `effectiveVaultName(vault)` from
the **cached** record (no disk read — for a fresh folder this is the folder name,
correct default), then after `openNote` loads settings, if the on-disk
`currentSettings.workspace` changed the effective name, re-stamp it (and refresh
the cache per AC-3). Snapshot the pre-attach `?ws` string into `prev` and restore
it verbatim on the openNote-failure rollback (replaces the current
`stampWorkspace(prev.vaultId)`/`unstampWorkspace()` branch, which assumed the
param was an id). `stampWorkspace` keeps using `replaceState` and preserving the
hash (unchanged).

Files: src/main.ts

## AC-9 — Per-vault session state stays keyed by the opaque id

Approach: No code change — `attachVaultNow` continues to key `loadRecency` /
`loadExpandedFolders` / `loadNoteList` and `currentVaultId` by `vault.id`
(FEAT-0059). Only the `?ws` param's meaning changes. Guard against regression with
a comment; the existing per-vault-session behavior is already covered by FEAT-0059
tests.

Files: (none — invariant preserved)

## AC-10 — No note bytes are written

Approach: The phase reads `.brulion.json` (via the existing `loadSettings`, which
this phase does not change) and writes only the browser-private vault cache
(IndexedDB) and the URL. No `.md` write path is touched. `setVaultWorkspace` writes
to idb-keyval only. Verified by inspection + the fact that no `note.ts`/`fs.ts`
write function is called from the new code.

Files: (none — invariant preserved)
