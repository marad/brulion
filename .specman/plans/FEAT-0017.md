# Sync plan — FEAT-0017 Clear Formatting completeness

Started: 2026-06-20T16:49:41.860Z
Snapshot state: new
Drift summary: 0 added (whole spec)

Add a parse-driven `clearFormatting` transform to `src/markdown-transforms.ts`
and route both entry points through it. The transform collects, from the markdown
syntax tree over the lines the action touches, the marker-character ranges to
delete:

- `EmphasisMark` (bold/italic `*`/`_`/`**`/`__`) and `CodeMark` whose parent is
  `InlineCode` (inline-code backticks) — deleted as-is. Driven by the parse tree,
  so nested marks (`**_x_**`) fully unwrap and a `*` inside `**` is unambiguous.
- `HeaderMark` and `QuoteMark`, and `ListMark` whose grandparent is `BulletList` —
  each deleted together with the single following space when present (the block
  prefix). Ordered-list `ListMark` (grandparent `OrderedList`) is left intact.
- Fenced-code `CodeMark` (parent `FencedCode`) is left intact (out of scope).

A shared helper `clearFormattingRanges(state, from, to)` returns the deletion
ranges over every line `[from, to)` touches (using the same end-line adjustment as
`setHeadingLines`, so a selection ending exactly at a line start doesn't pull in
the next line). `clearFormatting(state)` wraps it against the main selection and
returns a `TransactionSpec` of deletions, or `null` when there are none (the no-op
contract). Deletion ranges are non-overlapping (distinct marker nodes; a block
prefix sits before any inline mark on the line) so they compose in one change set.

Both callers reuse it:
- `src/context-menu.ts` — the "Clear formatting" item runs `clearFormatting`
  instead of `setHeadingLines(s, 0)`.
- `src/slash-commands.ts` — `/clear` removes its token (existing behavior) and
  merges in `clearFormattingRanges` for the token's line, so accepting `/clear`
  strips the line's markup too. The token range and the marker ranges are disjoint
  (the `/clear` text carries no markdown marks), so they apply together.
