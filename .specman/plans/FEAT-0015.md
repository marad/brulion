# Sync plan — FEAT-0015 Conflict resolution UX

Started: 2026-06-20T14:40:40.230Z
Snapshot state: new
Drift summary: 0 added (whole spec)

Close M4: turn the dead-end conflict state into a two-way choice. No new module
— two controller methods + a banner wired to them → **chisel** scope (file count
is inflated only by trivial HTML/CSS): `src/note-controller.ts`, `index.html`,
`src/main.ts`, `src/styles.css`, tests + an e2e.

**Proactive conflict in `refreshFromDisk` (controller).** Replace the Phase 2
"silently skip when unsafe" branch with: when the active note changed/deleted on
disk and the buffer is not safe to replace (`!safeToReplaceBuffer()` — unsaved
edits / save in flight) and not already in conflict, set `conflict = true` and
fire `opts.onConflict?.()`. The reactive path (doSave's stale-write guard, since
M1) already sets the same state — both converge. The no-edits path is unchanged
(reload / switch from FEAT-0014). The re-check after the read in the changed
branch (the race fix) now falls back to setting the conflict instead of bailing.

**Two resolution methods (controller, added to the interface), each on the
serialize queue, guarded by `if (!dir || !conflict) return`:**
- `resolveKeepMine(): Promise<void>` — re-base then overwrite: `const current =
  await statNote(dir, activeName)` (the file's present mtime, or null if deleted),
  `saveNote(dir, activeName, buffer, current)`. With `current` matching disk the
  guard passes; a deleted file (no existing handle) is re-created. On `saved`:
  adopt `lastModified`, clear `conflict`/`dirty`, re-list if the note was missing,
  `opts.onListChanged?.(notes, activeName)`, then `opts.onConflictResolved?.()`.
- `resolveTakeTheirs(): Promise<void>` — discard local, take disk: if `statNote`
  is null (deleted) clear `conflict`, re-list, `activate(pickActiveNote(notes,
  null))`; else `load(dir, activeName)` (clears conflict, adopts disk content +
  mtime, dirty=false) and `opts.onListChanged?.(notes, activeName)`. Then
  `opts.onConflictResolved?.()`.

**New option `onConflictResolved?: () => void`** so the UI can hide the banner
only when the conflict actually clears (not on unrelated list refreshes).

**UI.** `index.html`: a `<div id="conflict" hidden>` banner with a message and two
buttons (`#conflict-keep` "Keep my version", `#conflict-disk` "Use the version on
disk"). `main.ts`: `onConflict` reveals the banner; `onConflictResolved` hides it;
the two buttons call `resolveKeepMine` / `resolveTakeTheirs`. `styles.css`: style
the banner (clearly visible, not alarming). The old `#status`-message-on-conflict
is replaced by the banner; `#status` stays for create errors.

## AC-1 / AC-6 — Collision surfaces the conflict; nothing clobbered before choice
`refreshFromDisk` sets `conflict` + `onConflict` when an external change meets
unsaved edits; no write, no buffer change; autosave already suspended while
`conflict` (handleChange early-returns). Controller unit tests: dirty + external
change → `onConflict` called, `saveNote` not called, buffer intact.
Files: `src/note-controller.ts`, `src/note-controller.test.ts`

## AC-2 — Keep my version writes the buffer
`resolveKeepMine` re-bases on the current mtime and writes. Unit test: in
conflict with a moved disk mtime, `resolveKeepMine` → `saveNote` called with the
current mtime and the buffer content; conflict/dirty cleared; `onConflictResolved`
fired.
Files: `src/note-controller.ts`, `src/note-controller.test.ts`

## AC-3 — Use the version on disk reloads and drops local edits
`resolveTakeTheirs` (changed case) loads disk content, adopts mtime, clears
conflict. Unit test: buffer shows disk content, `dirty` false, a later save uses
the adopted mtime.
Files: `src/note-controller.ts`, `src/note-controller.test.ts`

## AC-4 — Editing works after either resolution
After resolve, `conflict` is false so `handleChange` schedules autosave again.
Unit tests: after keep-mine and after take-theirs, an edit triggers `saveNote`.
Files: `src/note-controller.ts`, `src/note-controller.test.ts`

## AC-5 — Deleted-under-edits is the same conflict
`refreshFromDisk` sets conflict for a deleted+dirty active note; `resolveKeepMine`
re-creates the file (no existing handle → write); `resolveTakeTheirs` switches off
it via `activate(pickActiveNote(notes, null))`. Unit tests for both resolutions of
a deleted-active conflict.
Files: `src/note-controller.ts`, `src/note-controller.test.ts`

## AC-7 — The conflict is modal until resolved (review correction)
Make the conflict modal instead of a dismissable banner:
- Controller: `switchTo`/`addNote`/`removeNote` become no-ops while `conflict`
  (guard at the top of each serialized op; `addNote` returns
  `{ok:false, reason}`). This is the logic-level guarantee that navigation can't
  silently abandon the conflicted buffer. With navigation blocked, the only path
  that clears a conflict via `load` is `resolveTakeTheirs`.
- Editor read-only: `editor.ts` gains `setEditorEditable(view, boolean)` backed by
  a `Compartment` reconfiguring `EditorState.readOnly` + `EditorView.editable`.
  `main.ts` toggles it false on `onConflict`, true on `onConflictResolved`.
- UI modality: `index.html` wraps `#conflict` in a full-screen `#conflict-backdrop`
  (fixed, semi-opaque, centered) that covers the list/editor/header; it does not
  dismiss on background click or Escape. `main.ts` toggles the backdrop's `hidden`.
- Controller unit tests: in conflict, `switchTo`/`addNote`/`removeNote` change
  nothing and leave `conflict` set (no `onConflictResolved`). e2e: the backdrop
  blocks sidebar clicks and the editor is read-only until a button is pressed.
Files: `src/note-controller.ts`, `src/note-controller.test.ts`, `src/editor.ts`,
`index.html`, `src/main.ts`, `src/styles.css`, `e2e/conflict.spec.ts`
