# Sync plan — FEAT-0049 syntax-highlight fenced code blocks

## Design

Chisel-scale: one new dependency, one small declarative file (a highlight style — no
logic/contract, so chisel not excavate), and a two-line wiring change in
`markdown-render.ts`.

Dependency: add `@codemirror/language-data` (the lazy `languages` resolver).

**Revised after code review:** a *global* `syntaxHighlighting(HighlightStyle)` would
recolor markdown prose, because prose and nested code share the highlight-tag
namespace (`@lezer/markdown` tags `Escape`→escape, `Comment`→comment,
`LinkTitle`→string — all visible/revealed prose the renderer leaves raw). So the
highlight is **scoped by range** to fenced blocks instead.

New `src/code-highlight.ts`:
- `collectCodeMarks(state, from, to): CodeMark[]` — iterate the syntax tree over
  `[from,to]`; for each `FencedCode` node, run `highlightTree(tree, classHighlighter,
  …, node.from, node.to)` and collect `{from,to,cls}` marks (stable `tok-*` classes).
  Only ranges inside a fenced block are ever marked, so prose is never touched (AC-4).
- `codeTokenTheme = EditorView.baseTheme({ ".tok-keyword": …, ".tok-string": …, … })`
  — a GitHub-light-ish palette keyed on the `tok-*` classes; only code-relevant
  classes are styled.

`src/markdown-render.ts`:
- import `{ languages }` from `@codemirror/language-data`, `{ collectCodeMarks,
  codeTokenTheme }` from `./code-highlight`.
- `markdown({ addKeymap: false, extensions: [Autolink], codeLanguages: languages })`.
- in `buildDecorations`, per visible range, push a `Decoration.mark({class})` for each
  `collectCodeMarks` result (deduped, skipping the frontmatter range).
- rebuild the render plugin on a `syntaxTree` change too (lazy language load reparses
  without a doc/viewport edit).
- add `codeTokenTheme` to the `markdownRendering` extension array.

Lazy: `languages` entries dynamic-import their parser on first use; Vite code-splits
them, the SW (FEAT-0029) caches them cache-first. Code-free notes load no grammar.

## Implementation steps (chisel — test before body)

1. e2e first (`e2e/code-highlight.spec.ts`, red): open a folder, type a fenced `js`
   block with a keyword/string/number, and assert the code box contains **multiple**
   highlight token spans of differing color (the CM highlight emits `<span>`s with
   `.tok-*`/style classes inside `.cm-code-block`); a second block with an unknown
   info string (e.g. ```` ```nope ````) shows **no** token spans (AC-2/AC-3); and the
   saved `start.md` bytes equal what was typed (AC-5). Wait past the lazy import with
   Playwright's auto-retrying expects.
2. Add the dependency, create `code-highlight.ts`, wire `markdown-render.ts`.
3. Unit (`src/code-highlight.test.ts`): mount an `EditorView` (happy-dom) with the
   real `markdownRendering` extension, set a doc with a ```js block, poll
   `syntaxTree` until the JS sub-tree appears (lazy load resolves in-process under
   Vite), and assert a JavaScript node nests over the block body (AC-1); assert an
   unknown info string yields no sub-language node (AC-3). If the lazy import proves
   unreliable under happy-dom, fall back to asserting AC-1/AC-3 in the e2e and keep a
   unit test only for the highlight style's tag coverage.
4. Confirm the existing markdown-render prose tests still pass unchanged (AC-4).
