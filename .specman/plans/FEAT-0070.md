## Implementation

Whole-spec plan (brand-new, same convention as FEAT-0068/FEAT-0069's plans).

### 1. `src/ui.ts` ŌĆö flat destination-path helper

- **`derivedFolderPaths(notes: string[], folders: string[]): string[]`** ŌĆö every
  unique folder prefix implied by `notes` (same prefix-walk `buildNoteTree`
  already does internally, just collecting the prefixes into a `Set` instead
  of building nodes) unioned with `folders` (the explicit empty-folder list),
  sorted with the same `localeCompare` convention used everywhere else. Root
  is **not** included here ŌĆö the picker (below) prepends the empty string
  `""` itself, since only it knows to special-case that as "(root)".

### 2. `src/move-picker.ts` ŌĆö new file, mirrors `quick-switcher.ts`

No shared overlay core exists today ŌĆö `quick-switcher.ts` (193 lines) and
`command-palette.ts` (176 lines) are two independent, near-identical
hand-rolled implementations (open/close, arrow/Enter/Esc keydown, highlight,
backdrop-click-to-close, `row()`/`render()`). This is a third instance of the
same shape; extracting a shared core is a legitimate follow-up but out of
scope for this phase (noted for the user, not done here ŌĆö see DECISIONS.md
entry to record this call).

- `MovePickerElements { backdrop, input, list, error }` ŌĆö same shape as
  `QuickSwitcherElements`.
- `mountMovePicker(els): MovePicker` with:
  - `open(folders: readonly string[], onPick: (destination: string) =>
    Promise<{ ok: boolean; reason?: string }>): void` ŌĆö `folders` is the
    caller-supplied list **including** `""` for root; `onPick` is bound fresh
    per open() call (the caller closes over whichever note/folder is being
    moved, so this module stays agnostic to what "moving" means).
  - Row label: `path === "" ? "(root)" : path`; filtering is a plain
    case-insensitive substring match on that label against the input value
    (not the full fuzzy `note-search.ts` scoring quick-switcher uses ŌĆö a
    folder list is small and flat; substring is simpler and sufficient).
  - Enter/click on a row calls `onPick(path)`; `!result.ok` shows
    `result.reason` in the error element (same pattern as quick-switcher's
    `setError`) and leaves the picker open; success closes it.
  - `close()`/`isOpen()`/`destroy()` ŌĆö same shape as `QuickSwitcher`.

### 5. `src/note-controller.ts` ŌĆö `moveFolder`

`renameActive`'s tail (note-controller.ts:706-761) is the template: flush ŌåÆ
`moveNote` ŌåÆ `rebaseMovedNote` (own outbound links, FEAT-0041) ŌåÆ relist ŌåÆ
`updateInboundLinks` (FEAT-0040, best-effort) ŌåÆ activate/announce.
`updateInboundLinks`/`rebaseMovedNote` are private closures over the
controller's own `notes` (note-controller.ts:464, 496) ŌĆö reused per relocated
file, not reimplemented.

- **Generalize `updateInboundLinks` to a list of moves**, not just one pair:
  `updateInboundLinksForMoves(folder, moves: {from,to}[], pathsBefore)` reads
  each *other* note once, folds every move's `rewriteLinksForRename` over its
  text in sequence (so a note linking to two different relocated files gets
  both rewrites in one read+write), writes back once if anything changed.
  `renameActive`'s existing call becomes `updateInboundLinksForMoves(dir,
  [{from, to: normalized.filename}], pathsBefore)` ŌĆö same behavior, generalized
  shape. (Rejected: N full-vault scans, one per moved file ŌĆö correct but
  O(moved files ├Ś vault size) redundant reads/writes for a folder with many
  notes.)
- **`moveFolder(fromPath, toPath): Promise<AddNoteResult>`**:
  1. Guard `!dir` / `conflict` (same messages as `addFolder`).
  2. Refuse when `toPath === fromPath` or `toPath.startsWith(fromPath + "/")`
     ŌĆö "Can't move a folder into itself or one of its own subfolders."
  3. If the active note is inside `fromPath` (`activeName === fromPath ||
     activeName.startsWith(fromPath + "/")`), `flushAndWait()` first (re-check
     `conflict` after, same guard `renameActive` has).
  4. `pathsBefore = new Set(notes)`; `toMove = notes.filter(p =>
     p.startsWith(fromPath + "/"))`; for each, `newPath = toPath +
     oldPath.slice(fromPath.length)`, `moveNote(dir, oldPath, newPath)` ŌĆö a
     per-file `exists`/`missing` result is skipped (left at its old path,
     not fatal to the rest), `moved` results collect into `moves: {from,
     to}[]` and clear `contentCache` for `oldPath`; run `rebaseMovedNote`
     per successfully-moved file (own outbound links).
  5. **Relocate now-empty subfolders too**: notes are the only thing that
     makes `toMove` non-empty, so an empty subfolder inside `fromPath` (e.g.
     `fromPath/empty-sub`, tracked only via `listFolders`, not `notes`) is
     never touched by step 4. After it, re-fetch `listFolders(dir)`, and for
     every remaining folder path starting with `fromPath + "/"` (now empty,
     since any notes it held were already moved out), `createFolder(dir,
     toPath + thatPath.slice(fromPath.length))` then `deleteFolder(dir,
     thatPath)` ŌĆö a trivial recreate-then-remove since there's no content
     left to lose.
  6. `notes = await listNotes(dir)`; `updateInboundLinksForMoves(dir, moves,
     pathsBefore)` (best-effort, try/catch, same reasoning as `renameActive`);
     `opts.onFoldersChanged?.(await listFolders(dir))`.
  7. If the active note was among `moves`, `activate(dir, itsNewPath)`; else
     `opts.onListChanged?.(notes, activeName)`.
  8. Return `{ ok: true }`.
