# Sync plan — FEAT-0059 Vault model and per-window identity

Started: 2026-06-25T05:13:13.397Z
Snapshot state: new
Drift summary: 0 added (whole spec)

Implemented via `/excavate` (new module `src/vaults.ts` + a reworked open/restore +
window-identity flow across `session.ts`, `ui.ts`, `main.ts`). Logical pieces:
- **vaults** (`src/vaults.ts`) — the persisted set of `{id,handle,name}` over
  idb-keyval: list / add (dedup by `isSameEntry`, generate opaque id) / get / remove,
  + one-time migration of the legacy `brulion:dir` handle.
- **per-vault session** (`src/session.ts`) — recency + expanded-folders keyed by vault
  id; a migration that copies the old global values onto the first vault. Sidebar
  width/collapse stay global; `saveFolder`/`loadFolder` removed (vaults replace them).
- **window identity** (`src/main.ts`) — read/stamp `?ws`, hold `currentVaultId`,
  `resolveStartupVault()`, `attachVault(vault)`; wire the open-folder + switch-folder
  paths to add-then-attach. `ui.ts` keeps the pick + permission/resume primitives.

## AC-1 (added) — Opening a folder records a vault and stamps `?ws`

Approach: `ui.openFolder` picks a handle and calls `onOpen(handle)`; the host's
handler `addVault(handle)` (vaults.ts) then `attachVault(vault)` — which stamps
`?ws=vault.id` via `history.replaceState` and opens it. `addVault` generates a short
opaque id (`crypto.randomUUID().slice(0,8)`, regen on collision) and persists the set.

Files: src/vaults.ts, src/ui.ts, src/main.ts

## AC-2 (added) — Reload re-attaches to the `?ws` vault

Approach: On load, `resolveStartupVault()` reads `?ws` from `location.search`, looks
it up via `getVault(id)`; if found, `restoreVault(vault, …)` (ui.ts) checks
permission and attaches. No picker.

Files: src/main.ts, src/ui.ts, src/vaults.ts

## AC-3 (added) — Two windows on different vaults stay independent

Approach: Property of `?ws` being per-window URL state (not a shared global handle):
each window's `resolveStartupVault` reads its own `?ws`. No shared "last opened" key
is consulted when `?ws` resolves. Covered by an e2e opening two pages with different
`?ws` and reloading each.

Files: src/main.ts

## AC-4 (added) — Re-picking a known folder reuses its vault

Approach: `addVault` iterates the stored vaults and `await handle.isSameEntry` each;
on a match it returns the existing vault (same id) instead of appending. The host
attaches the returned vault, so `?ws` matches the existing id.

Files: src/vaults.ts

## AC-5 (added) — Pre-M33 single handle migrates transparently

Approach: `migrateLegacyFolder()` (vaults.ts): if the set is empty and the legacy
`brulion:dir` handle exists, add it as the first vault and clear the legacy key; also
`migrateLegacySession(vaultId)` (session.ts) copies the old global recency/expanded
values to that vault's keys. Run once at startup before `resolveStartupVault`.

Files: src/vaults.ts, src/session.ts, src/main.ts

## AC-6 (added) — Fresh window with no `?ws` falls back to the most-recent vault

Approach: When `?ws` is absent/unknown, `resolveStartupVault` returns the
most-recently-used vault (the set is kept most-recent-first: attach moves the
attached vault to the front). Attaching stamps its `?ws`. Empty set → welcome screen
(unchanged).

Files: src/main.ts, src/vaults.ts

## AC-7 (added) — Re-granting permission still works per window

Approach: `restoreVault` reuses the existing query/request-permission + resume-button
flow (the old `restoreFolder` logic, parameterized on a specific vault handle): if
`queryPermission` isn't granted, reveal the resume button wired to `requestPermission`
then attach.

Files: src/ui.ts, src/main.ts

## AC-8 (added) — `?ws` uses replaceState

Approach: `stampWorkspace(id)` builds a URL preserving the existing hash and sets
`?ws=id`, written with `history.replaceState` only — never `pushState`/`location`
push.

Files: src/main.ts

## AC-9 (added) — No note bytes written

Approach: Property of the design — vaults/ids/`?ws`/per-vault session keys are all
idb-keyval + URL; no `FileSystemFileHandle` write. Verified by an e2e asserting the
folder's `.md` files are untouched across open/reload/switch.

Files: (none — property of the storage-only change)

## AC-10 (added) — Recency and expanded-folders are per-vault

Approach: `session.ts` recency/expanded getters+setters take a `vaultId` and key as
`brulion:recency:<id>` / `brulion:expanded-folders:<id>`. `main.ts` passes
`currentVaultId` (set in `attachVault`) and reloads recency + expanded on each attach
so a switch shows the new vault's state. Sidebar width/collapse keys unchanged
(global). Legacy values migrated per AC-5.

Files: src/session.ts, src/main.ts
