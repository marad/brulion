# Sync plan — FEAT-0012 Create and delete notes

Started: 2026-06-20T12:17:12.710Z
Snapshot state: new
Drift summary: 0 added (whole spec)

Add the two mutating verbs on top of FEAT-0010/0011. No new module: the
controller gains `addNote`/`removeNote`, `ui.ts` grows a new-note form helper and
a per-row delete button, `main.ts` wires them (delete via `window.confirm`).

Controller refactor: extract a private `activate(name)` (load + persist active +
`onListChanged`) from the existing switch tail, so switch/create/delete share one
re-point path. All three run inside the existing `serialize()` queue.
- `addNote(name)`: `normalizeNoteName` → on invalid return `{ok:false, reason}`;
  `createNote(dir, filename)` → on `exists` return `{ok:false, reason}`; else
  re-list, `flushAndWait()` the open note, `activate(filename)`, return `{ok:true}`.
- `removeNote(name)`: if `name === activeName`, cancel autosave + clear `dirty`
  (don't resurrect the file we're deleting); else `flushAndWait()` the open note.
  `deleteNote(dir, name)`, re-list. If the active note was deleted,
  `activate(pickActiveNote(notes, null))`; else just `onListChanged(notes,
  activeName)`.

UI:
- `index.html`: wrap the sidebar as `<aside id="sidebar" hidden>` containing a
  `<form id="new-note">` (text input) and `<div id="note-list">`; reveal it when a
  folder opens.
- `ui.ts`: `renderNoteList` takes a handlers object `{onSelect, onDelete}` and adds
  a delete button per row. New `wireNewNote(form, input, onCreate)` submits the
  trimmed value and clears the input.
- `main.ts`: reveal `#sidebar` in `onListChanged`; wire the new-note form to
  `controller.addNote` (showing `reason` in `#status` on failure, clearing it on
  success); wire row delete to a `window.confirm` then `controller.removeNote`.
- `styles.css`: style the new-note input and the delete button (hover-revealed).

## AC-1 / AC-2 — Create a named note, open it, flush the previous
`addNote` validates, creates, re-lists, flushes the open note via the shared
guarded save, then `activate(filename)`. Controller unit test: valid name →
`createNote` called, editor shows empty, `onListChanged` active = new file;
dirty previous note flushed first (assert save ordering).
Files: `src/note-controller.ts`, `src/note-controller.test.ts`, `src/ui.ts`, `src/ui.test.ts`, `src/main.ts`, `index.html`, `src/styles.css`

## AC-3 / AC-4 — Refuse duplicates and invalid names with a message
`addNote` returns `{ok:false, reason}` for `normalizeNoteName` failures and for a
`createNote` `exists` result; the editor/active note are untouched. `main.ts`
shows `reason` in `#status`. Unit tests for both branches; e2e asserts the
message and that no switch happened.
Files: `src/note-controller.ts`, `src/note-controller.test.ts`, `src/main.ts`

## AC-5 — Delete asks for confirmation
Row delete buttons call `onDelete(name)`; `main.ts` gates `removeNote` behind
`window.confirm`. Declining is a no-op. e2e dismisses the dialog and asserts the
note remains.
Files: `src/ui.ts`, `src/ui.test.ts`, `src/main.ts`

## AC-6 / AC-8 — Confirmed deletion removes file + row; non-active leaves editor
`removeNote` deletes the file and re-lists. Deleting a non-active note keeps
`activeName` and just re-renders the list. Controller unit tests + e2e.
Files: `src/note-controller.ts`, `src/note-controller.test.ts`, `src/ui.ts`

## AC-7 — Deleting the active note switches to another
`removeNote` drops the active note's pending edits, deletes, re-lists, and
`activate(pickActiveNote(notes, null))` — `start.md` / first / empty `start`
buffer when the folder is now empty. Controller unit tests cover all three
fallbacks; e2e covers delete-active → next note.
Files: `src/note-controller.ts`, `src/note-controller.test.ts`
