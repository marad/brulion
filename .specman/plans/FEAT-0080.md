## AC-1, AC-2, AC-3 — The Workspace name field (render, seed, placeholder, emit)

Approach: In `src/settings-modal.ts`, add a "Workspace name" text input mirroring
the existing `journalInput` (FEAT-0062): `type=text`, class `settings-workspace`,
anti-autofill attrs, an `input` listener `emit({ workspace: value })` (AC-3, the
existing modal→`onChange`→`updateSettings`→`saveSettings` pipeline), and a
`labeledRow("Workspace name", …)` appended to the sections list. In `seed()`, set
`workspaceInput.value` from `s.workspace` with the same caret-guard as
`journalInput` (only reassign when it differs), and set `workspaceInput.placeholder
= handlers.getFolderName()` so an empty field shows the folder-name default (AC-2).
AC-1 is the seeded value; the modal stays state-free (it already exposes
`getFolderName`, FEAT-0054).

Files: src/settings-modal.ts, src/settings-modal.test.ts

## AC-4, AC-5 — Live `?ws` + cached vault name on change

Approach: In `src/main.ts` `updateSettings`, after the existing apply/persist, when
the patch changes `workspace` and a vault is attached (`"workspace" in patch &&
currentVaultId`), refresh the live identity via the FEAT-0079 helpers: `await
markVaultAttached(currentVaultId, currentSettings.workspace)` (caches the new name)
and `stampWorkspace(effectiveVaultName({ name: settingsDir?.name ?? "", workspace:
currentSettings.workspace }))` (re-stamps the URL to the new effective name).
Clearing the field (empty `workspace`) makes `effectiveVaultName` fall back to the
folder name, so `?ws` and the cache return to it (AC-5). Gated on `"workspace" in
patch` so other setting changes (vim/theme/font) are unaffected. No parallel
implementation — the same helpers the attach path uses.

Files: src/main.ts

## AC-6 — Only `.brulion.json` is written

Approach: The field rides `saveSettings` (writes `.brulion.json` only) and the live
update writes the browser-private vault cache (`markVaultAttached` → idb) and the
URL. No `.md` write path is touched. Verified by inspection + an e2e md-count
assertion.
