# Sync plan — FEAT-0040 Update references when a note is renamed

Started: 2026-06-23T09:13:25.874Z
Snapshot state: new
Drift summary: 0 added (whole spec)

## Approach

One new pure module for the byte-mutating rewrite, plus an inbound-link pass added
to the existing `renameActive`, plus the confirm wiring in `main.ts`.

### New module: `src/link-rewrite.ts` (AC-1..AC-8) — pure, no DOM/FSA/CM

Detects every link in one note's text that points at `oldPath` and rewrites it to
point at `newPath`. Reuses `resolveNotePath`/`resolveWikilink` (detection) and
`shortestLinkText` (bare-vs-full wikilink form) — single source of truth.

Exports:

- `relativeLink(fromNote: string, target: string): string` — POSIX relative path
  from `fromNote`'s folder to `target`. Common-prefix of the two dir segment lists,
  `..` for each remaining `fromDir` segment, then the remaining target segments +
  filename. Invariant: `resolveNotePath(fromNote, relativeLink(fromNote, t)) === t`
  (round-trip, AC-1/AC-2). Same-folder → bare filename (no `./`).
- `markdownDest(rel: string): string` — wrap `rel` in `<…>` iff it contains a space
  or `(`/`)`, else return it bare (AC-3). (CommonMark angle-bracket destination;
  our paths never contain `<`/`>` — `normalizeNoteName` rejects them.)
- `rewriteLinksForRename(args: { text, notePath, oldPath, newPath, pathsBefore,
  pathsAfter }): string | null` — collect edits `{ from, to, insert }`, apply them
  right-to-left (descending `from`) so offsets stay valid, return the new text or
  `null` if no edit.

Detection passes:
1. **Markdown links** via the markdown grammar. Use `@lezer/markdown`'s `parser`
   (pure; same CommonMark grammar the renderer uses) — `parser.parse(text)`, walk
   the tree with `tree.cursor()`, for each `Link` node (NOT `Image`) find its
   `URL` child, slice the dest. Skip `isExternalLink(dest)`. If
   `resolveNotePath(notePath, dest) === oldPath`, emit an edit over the URL node's
   `[from,to)` with `markdownDest(relativeLink(notePath, newPath))` (AC-1/2/3, AC-8
   excludes images + external).
2. **Wikilinks** via `WIKILINK_RE` (the renderer's scan). For each match, trim the
   target; `resolveWikilink(target, pathsBefore).resolved === oldPath`? If so, emit
   an edit over the *target* sub-range (`from+2 .. from+2+targetLen`, i.e. before
   any `|alias`) — preserving brackets and alias. New target:
   - slashed (target contains `/`) → `displayName(newPath)` (AC-7);
   - bare → `shortestLinkText(newPath, pathsAfter)` — bare basename when unique,
     full path only when the move would make it ambiguous (AC-4/5/6). When the new
     target equals the current trimmed target (a folder move keeping a unique
     basename), emit no edit (AC-5 → may yield `null`).

`pathsBefore` is used for detection (so a bare wikilink still recognized though
`oldPath` is gone post-move); `pathsAfter` for the new form.

### Controller: inbound-link pass in `renameActive` — `src/note-controller.ts`

After the existing `moveNote` success and BEFORE/around the `activate`, add the
pass (AC-9..AC-12), still inside the same `serialize` slot:

1. Add `confirmLinkUpdate?: (affected: string[]) => Promise<boolean> | boolean` to
   `NoteControllerOptions`.
2. Capture `const from = activeName` at the top (already the active path).
   `const pathsBefore = new Set(notes)` (pre-move listing). Refresh
   `notes = await listNotes(dir)`; `const pathsAfter = new Set(notes)`.
3. `await activate(dir, to)` — editor follows immediately (unchanged FEAT-0034).
4. Build edits: for each `path of notes` with `path !== to`, `readNote(dir, path)`,
   run `rewriteLinksForRename`; if non-null collect `{ path, text, lastModified }`.
   (Iterate the post-move listing; `from` is gone, `to` is excluded — the renamed
   note never rewrites itself.)
5. If edits non-empty: `const ok = opts.confirmLinkUpdate ? await
   opts.confirmLinkUpdate(edits.map(e => e.path)) : true`. If `ok`, for each edit
   `await saveNote(dir, e.path, e.text, e.lastModified)` — a `conflict` result is
   skipped (AC-12, never clobber). If declined, skip all (AC-10 — rename stands).
6. Return `{ ok: true }`.

The renamed note's own bytes are untouched (it is excluded from the scan). No
inbound links → no confirm, no extra write (AC-11).

### UI wiring — `src/main.ts`

Pass `confirmLinkUpdate` into `createNoteController`: a blocking `window.confirm`
naming the affected notes' display names + count
(`"Update links to \"<old>\" in N note(s)?\n\n<list>"`). Decline → links dangle.
(`renameActive` is still called from `mountNoteIdentity`'s rename callback,
unchanged.)
