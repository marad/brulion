# Sync plan — FEAT-0073 In-app dialogs replace native confirm/prompt/alert

Started: 2026-07-15T05:55:10.445Z
Snapshot state: new
Drift summary: 0 added (whole spec)

## Implementation

Whole-spec plan (brand-new). One new module, mirroring `move-picker.ts`'s
`mount(els)` shape, wired into `main.ts` in place of every M35-introduced
native dialog call. No controller/file-system changes.

### 1. `src/dialog.ts` (new)

```ts
export interface DialogElements {
  backdrop: HTMLElement
  dialog: HTMLElement
  message: HTMLElement
  input: HTMLInputElement
  cancelButton: HTMLButtonElement
  confirmButton: HTMLButtonElement
}

export interface Dialog {
  confirm(message: string, confirmLabel?: string): Promise<boolean>
  prompt(message: string, initialValue?: string, confirmLabel?: string): Promise<string | null>
  alert(message: string): Promise<void>
  destroy(): void
}

export function mountDialog(els: DialogElements): Dialog
```

One shared `open(mode, message, initialValue, confirmLabel)` internal
function drives all three public methods, toggling `input.hidden` and
`cancelButton.hidden` per mode:

- **alert** — input hidden, cancel hidden, confirm button reads "OK";
  confirm click / Escape / backdrop click / Enter all just dismiss.
- **confirm** — input hidden, cancel shown, confirm button reads
  `confirmLabel` (default "OK"); confirm click or Enter resolves `true`,
  cancel/Escape/backdrop resolves `false`.
- **prompt** — input shown and pre-filled with `initialValue` then focused
  (cursor at the end, mirroring `quick-switcher.ts`'s `open(initialQuery)`);
  confirm click or Enter (in the input) resolves `input.value`,
  cancel/Escape/backdrop resolves `null`.

Same scaffolding as `move-picker.ts`: a `generation` counter so a superseded
open auto-resolves the previous call's promise with its cancel value instead
of leaving it dangling; `restoreFocus` captured on open, refocused on close;
`backdrop.hidden` toggles visibility; a document-level `keydown`
(Escape) and the backdrop's own `click` (only when `event.target ===
backdrop`) both close. `destroy()` removes listeners (test/teardown parity
with `move-picker.ts`/`mountMovePicker`).

### 2. `index.html` — new markup

A new backdrop, same family as `#move-backdrop`/`#conflict-backdrop`,
inserted after `#move-backdrop`:

```html
<div id="dialog-backdrop" hidden>
  <div id="dialog" role="alertdialog" aria-modal="true" aria-label="Confirm">
    <p id="dialog-message"></p>
    <input id="dialog-input" type="text" aria-label="Name" autocomplete="off" hidden />
    <div class="dialog-actions">
      <button id="dialog-cancel" type="button">Cancel</button>
      <button id="dialog-confirm" type="button">OK</button>
    </div>
  </div>
</div>
```

### 3. `src/main.ts` — wire it, replace every M35 native call

- Grab the five elements near the other backdrop lookups and
  `const dialog = mountDialog({ ... })` alongside `const movePicker =
  mountMovePicker({ ... })`.
- `onDelete`: `window.confirm(...)` → `void dialog.confirm(message,
  "Delete").then((ok) => { if (ok) void controller.removeNote(name) })`.
- `onDeleteFolder`: same shape, folder message, `"Delete"` label.
- `promptNewFolder`: `window.prompt("New folder name:")` →
  `dialog.prompt("New folder name:")`; `window.alert(result.reason)` →
  `void dialog.alert(result.reason ?? "Could not create it.")`.
- `promptRenameTo`: `window.prompt("Rename to:", currentLeaf)` →
  `dialog.prompt("Rename to:", currentLeaf)`; failure `window.alert` →
  `void dialog.alert(...)`, same fallback message.
- `onDropNote`/`onDropFolder`: `window.alert(result.reason)` →
  `void dialog.alert(result.reason ?? "Could not move it there.")`.
- `openNotePath`'s "doesn't exist yet — create it?" `window.confirm` is
  **unchanged** (out of scope, per the spec).

### 4. `src/styles.css` — extend the existing modal-family selector groups

Add `#dialog-backdrop` alongside `#conflict-backdrop, #settings-backdrop`
(the base rule, its `@starting-style` block, and its `[hidden]` rule) and
`#dialog` alongside `#conflict, .settings-dialog` (same three places) so the
new dialog inherits the exact same backdrop/motion/focus-restore behavior
with zero new transition rules. Add `.dialog-actions` next to
`.conflict-actions` and `#dialog button` next to `#conflict button` for
layout/hover, plus one small `#dialog` sizing rule (`width: min(24rem,
100%)` — a confirm/prompt box, not the conflict diff's 60rem) and a
`#dialog-input` rule mirroring `#move-input`'s existing text-input styling.
