# Sync plan — FEAT-0064 Reveal code-fence markers on edit

Implemented via `/chisel` — a small change in `src/markdown-render.ts` (the FEAT-0016
block field). No new module.

## AC-1 (added) — caret inside reveals the fences

Approach: in `blockSyntaxRanges`, read `state.selection.main`; in the `FencedCode`
branch, compute `reveal = sel.from < closeLine.to && sel.to > openLine.from` and skip
the two `hidden.push` fence ranges when `reveal` is true (the fence text stays
visible). Strict overlap, matching the link/Mermaid reveals.

Files: src/markdown-render.ts

## AC-2 (added) — leaving re-hides

Approach: `blockRenderingField.update` rebuilds on `tr.docChanged || tr.selection` (it
currently rebuilds only on `docChanged`), so moving the caret out recomputes the field
and re-hides the fences.

Files: src/markdown-render.ts

## AC-3 (added) — info string editable in place

Approach: with the fence text un-hidden (and so not in the atomic/hidden set), the
caret rests on it and edits apply normally; leaving rebuilds and re-renders. No extra
code beyond AC-1/AC-2.

Files: src/markdown-render.ts

## AC-4 (added) — code-box styling persists while revealed

Approach: the `lines.push({ cls: "cm-code-block" + edge })` loop runs regardless of
`reveal` — only the `hidden` fence ranges are conditional. So the box styling stays.

Files: src/markdown-render.ts

## AC-5 (added) — a block the selection is outside renders fenceless (unchanged)

Approach: when `reveal` is false the two fence-hide ranges are pushed exactly as
before — no regression to FEAT-0016.

Files: src/markdown-render.ts

## AC-6 (added) — Mermaid fence revealed with its source

Approach: property — Mermaid's own field already reveals the raw block on selection
(FEAT-0056); the same selection now makes `blockSyntaxRanges` reveal the fence, so
```` ```mermaid ```` shows. No Mermaid-specific code. Covered by an e2e.

Files: (none — falls out of the shared block field)
